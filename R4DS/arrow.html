22  Arrow

22.1 はじめに

CSV ファイルは、人間が読みやすいように設計されています 。これらは非常に単純であり、世の中のあらゆるツールで読み取ることができるため、優れた交換フォーマットと言えます 。しかし、CSV ファイルはあまり効率的ではありません。R にデータを読み込むために、かなりの作業が必要になります 。この章では、ビッグデータシステムで広く使用されている、オープン標準に基づいた強力な代替手段である Parquet フォーマットについて学びます 。

Parquet ファイルと組み合わせて、大規模なデータセットの効率的な分析と転送のために設計されたマルチ言語ツールボックスである Apache Arrow を使用します 。Apache Arrow は arrow パッケージを介して利用します。このパッケージは dplyr バックエンドを提供しており、慣れ親しんだ dplyr 構文を使用して、メモリ容量を超える規模のデータセットを分析できるようになります 。追加の利点として、Arrow は非常に高速です。その例は、この章の後半で確認します 。

arrow と dbplyr はどちらも dplyr バックエンドを提供しているため、どちらを使うべきか迷うかもしれません 。多くの場合、データがすでにデータベース内にあるか、Parquet ファイルとして存在しているため、選択の余地なくそのまま扱うことになるでしょう 。しかし、自分自身のデータ（例えば CSV ファイル）から始める場合は、それをデータベースにロードするか、Parquet に変換するかを選択できます 。一般的に、どちらが最適かを判断するのは難しいため、分析の初期段階では両方を試してみて、自分に最も合った方を選ぶことをお勧めします 。

22.1.1 事前準備

この章では引き続き tidyverse（特に dplyr）を使用しますが、大規模なデータを扱うために特別に設計された arrow パッケージと組み合わせて使用します 。

library(tidyverse)
library(arrow)

この章の後半では、Arrow と DuckDB の連携についても確認するため、dbplyr と duckdb も必要になります 。

library(dbplyr, warn.conflicts = FALSE)
library(duckdb)
#> Warning: package 'duckdb' was built under R version 4.5.3
#> Loading required package: DBI

22.2 データの取得

これらのツールにふさわしいデータセットから始めましょう。シアトル公立図書館のアイテム貸出データセットで、data.seattle.gov/Community/Checkouts-by-Title/tmmm-ytt6 からオンラインで入手できます 。このデータセットには 41,389,465 行が含まれており、2005年4月から2022年10月まで、各本が毎月何回貸し出されたかを示しています。

22.3 データセットを開く

まずはデータを見てみましょう 。このファイルは 9 GB もあり、全体をメモリにロードしたくないほど巨大です 。目安として、通常はデータのサイズの少なくとも 2 倍のメモリが必要ですが、多くのノートパソコンは 16 GB が上限です 。つまり、read_csv() を避け、代わりに arrow::open_dataset() を使用します 。

seattle_csv <- open_dataset(
  sources = "data/seattle-library-checkouts.csv",
  col_types = schema(ISBN = string()),
  format = "csv"
)

このコードを実行すると何が起こるでしょうか 。open_dataset() は数千行をスキャンして、データセットの構造を把握します 。最初の 80,000 行において ISBN 列に空白が含まれているため、Arrow がデータ構造を解明できるように列型を指定する必要があります 。open_dataset() によるスキャンが終わると、見つかった情報を記録して停止します 。それ以降の行は、具体的にリクエストされたときにのみ読み込まれます 。

seattle_csv を表示すると、このメタデータを確認できます 。

22.4 Parquet フォーマット

このデータを扱いやすくするために、Parquet ファイルフォーマットに切り替え、複数のファイルに分割しましょう 。以下のセクションでは、まず Parquet とパーティショニング（分割）について紹介し、学んだ内容をシアトル図書館のデータに適用します 。

22.4.1 Parquet の利点

CSV と同様に、Parquet は矩形データに使用されます。しかし、どのファイルエディタでも読めるテキスト形式ではなく、ビッグデータのニーズに合わせて特別に設計されたカスタムバイナリ形式です 。これは以下のことを意味します。

  • Parquet ファイルは通常、同等の CSV ファイルよりもサイズが小さい 。Parquet はファイルサイズを抑えるために効率的なエンコーディングを採用しており、ファイルの圧縮をサポートしています 。ディスクからメモリに移動するデータが少なくなるため、Parquet ファイルの高速化に役立ちます 。
  • Parquet ファイルは豊富な型システムを持っている 。Section 7.3 で説明したように、CSV ファイルは列型に関する情報を提供しません 。例えば、CSV リーダーは "08-10-2022" を文字列としてパースすべきか日付としてパースすべきかを推測しなければなりません 。対照的に、Parquet ファイルはデータとともに型を記録する方法でデータを保存します 。
  • Parquet ファイルは「列指向（カラム指向）」である 。これは、R のデータフレームと同様に、列ごとにデータが整理されていることを意味します 。行ごとに整理されている CSV ファイルと比較して、通常、データ分析タスクのパフォーマンスが向上します 。
    • Parquet ファイルは「チャンク（塊）」に分割されている。これにより、ファイルの異なる部分に対して同時に作業を行うことができ、運が良ければ一部のチャンクを完全にスキップすることも可能です 。

Parquet ファイルには大きな欠点が 1 つあります。「人間が読める形式」ではなくなっていることです。つまり、readr::read_file() を使って Parquet ファイルを見ても、文字化けしたデータの塊が見えるだけです 。

22.4.2 パーティショニング

データセットが大きくなるにつれ、すべてのデータを単一のファイルに保存することはますます困難になります。そのため、大規模なデータセットを多くのファイルに分割することがしばしば有効です 。この構造化を賢く行うと、多くの分析でファイルの一部しか必要としないため、パフォーマンスの劇的な向上につながります 。

データセットをどのようにパーティショニング（分割）すべきかについて、厳格なルールはありません。結果はデータ、アクセスパターン、およびデータを読み取るシステムに依存します 。自分の状況に最適なパーティショニングを見つけるには、いくつかの実験が必要になるでしょう 。大まかなガイドラインとして、Arrow は 20 MB 未満および 2 GB を超えるファイルを避け、パーティションが 10,000 ファイルを超えないようにすることを推奨しています 。

また、フィルター処理に使用する変数でパーティショニングを試みるべきです 。後ほど説明するように、これにより Arrow は関連するファイルのみを読み取ることで、多くの作業をスキップできるようになります 。

22.4.3 シアトル図書館データの書き換え

これらのアイデアをシアトル図書館のデータに適用して、実際にどのように機能するかを確認しましょう 。一部の分析では最近のデータのみを見たい場合があり、年ごとにパーティショニングすると適度なサイズの 18 個のチャンクが得られるため、CheckoutYear でパーティショニングします 。

データを書き換えるには、dplyr::group_by() を使ってパーティションを定義し、arrow::write_dataset() を使ってパーティションをディレクトリに保存します 。write_dataset() には 2 つの重要な引数があります。ファイルを作成するディレクトリと、使用するフォーマットです 。

pq_path <- "data/seattle-library-checkouts"
seattle_csv |>
  group_by(CheckoutYear) |>
  write_dataset(path = pq_path, format = "parquet")

この実行には約 1 分かかります 。後ほど確認するように、これは初期投資であり、将来の操作を格段に速くすることで報われます 。作成したものを確認してみましょう。

tibble(
  files = list.files(pq_path, recursive = TRUE),
  size_MB = file.size(file.path(pq_path, files)) / 1024^2
)
#> # A tibble: 18 × 2
#>   files                            size_MB
#>   <chr>                              <dbl>
#> 1 CheckoutYear=2005/part-0.parquet    109.
#> 2 CheckoutYear=2006/part-0.parquet    164.
#> 3 CheckoutYear=2007/part-0.parquet    177.
#> 4 CheckoutYear=2008/part-0.parquet    194.
#> 5 CheckoutYear=2009/part-0.parquet    214.
#> 6 CheckoutYear=2010/part-0.parquet    222.
#> # ℹ 12 more rows

単一の 9 GB の CSV ファイルが 18 個の Parquet ファイルに書き換えられました 。ファイル名は、Apache Hive プロジェクトで使用されている「自己記述的（self-describing）」な慣習に従っています 。

【Geminiの補足】: Hive スタイルのパーティショニングでは、フォルダ名が key=value という形式になります。例えば、CheckoutYear=2005 というフォルダには、貸出年が 2005 年のデータのみが格納されます。Arrow などのツールは、フォルダ名を見るだけで中身を判断できるため、不要なフォルダ（年）を読み飛ばすことができます。

各ファイルは 100 〜 300 MB で、総サイズは約 4 GB となり、元の CSV ファイルの半分強のサイズになりました 。Parquet ははるかに効率的なフォーマットであるため、これは期待通りの結果です 。

22.5 Arrow で dplyr を使用する

Parquet ファイルを作成したので、これらを再度読み込む必要があります 。再び open_dataset() を使いますが、今回はディレクトリを指定します 。

seattle_pq <- open_dataset(pq_path)

これで dplyr パイプラインを書くことができます 。例えば、過去 5 年間に各月で貸し出された本の総数をカウントできます。

query <- seattle_pq |>
  filter(CheckoutYear >= 2018, MaterialType == "BOOK") |>
  group_by(CheckoutYear, CheckoutMonth) |>
  summarize(TotalCheckouts = sum(Checkouts)) |>
  arrange(CheckoutYear, CheckoutMonth)

Arrow データに対して dplyr コードを書くことは、概念的には dbplyr (Chapter 21) と似ています。dplyr コードを書くと、それが Apache Arrow C++ ライブラリが理解できるクエリに自動的に変換され、collect() を呼び出したときに実行されます 。

query オブジェクトを表示すると、実行時に Arrow が何を返すかについての情報を少し確認できます。

query
#> FileSystemDataset (query)
#> CheckoutYear: int32
#> CheckoutMonth: int64
#> TotalCheckouts: int64
#> 
#> * Grouped by CheckoutYear
#> * Sorted by CheckoutYear [asc], CheckoutMonth [asc]
#> See $.data for the source Arrow object

そして、collect() を呼び出すことで結果を取得できます。

query |> collect()
#> # A tibble: 58 × 3
#> # Groups:   CheckoutYear [5]
#>   CheckoutYear CheckoutMonth TotalCheckouts
#>          <int>         <int>          <int>
#> 1         2018             1         355101
#> 2         2018             2         309813
#> 3         2018             3         344487
#> 4         2018             4         330988
#> 5         2018             5         318049
#> 6         2018             6         341825
#> # ℹ 52 more rows

dbplyr と同様に、Arrow は一部の R 式しか理解できないため、普段書いているコードと全く同じには書けない場合があります 。しかし、サポートされている操作や関数のリストはかなり広範囲であり、増え続けています 。現在サポートされている関数の完全なリストは ?acero で確認してください 。

22.5.1 パフォーマンス

CSV から Parquet に切り替えたことによるパフォーマンスへの影響を簡単に見てみましょう 。まず、データが単一の大きな CSV として保存されている場合に、2021年の各月に貸し出された本の数を計算するのにかかる時間を測定します。

seattle_csv |>
  filter(CheckoutYear == 2021, MaterialType == "BOOK") |>
  group_by(CheckoutMonth) |>
  summarize(TotalCheckouts = sum(Checkouts)) |>
  arrange(desc(CheckoutMonth)) |>
  collect() |>
  system.time()
#>    user  system elapsed 
#>   20.26    9.31   24.39

次に、シアトル図書館の貸出データが 18 個の小さな Parquet ファイルにパーティショニングされた新しいバージョンのデータセットを使用してみましょう。

seattle_pq |>
  filter(CheckoutYear == 2021, MaterialType == "BOOK") |>
  group_by(CheckoutMonth) |>
  summarize(TotalCheckouts = sum(Checkouts)) |>
  arrange(desc(CheckoutMonth)) |>
  collect() |>
  system.time()
#>    user  system elapsed 
#>    0.22    0.19    0.21

約 100 倍のパフォーマンス向上が見られますが、これは「複数ファイルへのパーティショニング」と「個々のファイルのフォーマット」という 2 つの要因によるものです。

  • パーティショニングは、このクエリが CheckoutYear == 2021 を使ってデータをフィルター処理しているため、パフォーマンスを向上させます。Arrow は 18 個の Parquet ファイルのうち 1 つだけを読み取ればよいことを認識できるほどスマートです 。
  • Parquet フォーマットは、データをバイナリ形式で保存し、メモリに直接読み取れるようにすることでパフォーマンスを向上させます 。列単位のフォーマットと豊富なメタデータにより、Arrow はクエリで実際に使用される 4 つの列（CheckoutYear、MaterialType、CheckoutMonth、Checkouts）のみを読み取るだけで済みます 。

この圧倒的なパフォーマンスの差こそが、巨大な CSV を Parquet に変換する価値がある理由です 。

22.5.2 Arrow で DuckDB を使用する

Parquet と Arrow にはもう 1 つの利点があります。arrow::to_duckdb() を呼び出すことで、Arrow データセットを DuckDB データベース (Chapter 21) に非常に簡単に変換できることです。

seattle_pq |>
  to_duckdb() |>
  filter(CheckoutYear >= 2018, MaterialType == "BOOK") |>
  group_by(CheckoutYear) |>
  summarize(TotalCheckouts = sum(Checkouts)) |>
  arrange(desc(CheckoutYear)) |>
  collect()
#> Warning: Missing values are always removed in SQL aggregation functions.
#> Use `na.rm = TRUE` to silence this warning
#> This warning is displayed once every 8 hours.
#> # A tibble: 5 × 2
#>   CheckoutYear TotalCheckouts
#>          <int>          <dbl>
#> 1         2022        2431502
#> 2         2021        2266438
#> 3         2020        1241999
#> 4         2019        3931688
#> 5         2018        3987569

to_duckdb() の素晴らしい点は、この転送にメモリのコピーを伴わないことです。これは、ある計算環境から別の計算環境へのシームレスな移行を可能にするという Arrow エコシステムの目標を体現しています 。

22.6 まとめ

この章では、ディスク上の大規模なデータセットを扱うための dplyr バックエンドを提供する arrow パッケージを体験しました 。Arrow は CSV ファイルでも動作しますが、データを Parquet に変換すれば格段に速くなります 。Parquet は、現代のコンピュータでのデータ分析のために特別に設計されたバイナリデータ形式です 。CSV と比較して Parquet ファイルを扱えるツールははるかに少ないですが、パーティショニング、圧縮、および列指向の構造により、分析の効率が大幅に向上します 。

次は、最初の非矩形データソースについて学びます。これは tidyr パッケージが提供するツールを使用して扱います 。主に JSON ファイルからのデータを扱いますが、その一般原則はソースに関係なくツリー状のデータに適用されます 。
