4  ワークフロー：コードスタイル

良いコーディングスタイルは、正しい句読点のようなものです。それがなくても何とかなりますが、butitsuremakesthingseasiertoread（あるとはるかに読みやすくなります） 。 プログラミングを始めたばかりであっても、コードスタイルを整えることは非常に良いアイデアです 。 一貫したスタイルを使用することで、他の人（未来の自分自身も含む！）があなたの作業を読みやすくなり、誰かに助けを求める際には特に重要になります 。 この章では、本書全体で使用されている「tidyverse スタイルガイド」の最も重要なポイントを紹介します 。 コードのスタイリングは最初は少し面倒に感じるかもしれませんが、練習すればすぐに第2の天性（無意識にできること）になるでしょう 。 さらに、Lorenz Walthert による styler パッケージのように、既存のコードを素早く再スタイリングするための優れたツールも存在します 。 install.packages("styler") でインストールした後は、RStudio のコマンドパレットから使うのが簡単です 。 コマンドパレットでは、RStudio のあらゆる組み込みコマンドや、パッケージが提供する多くのアドインを使用できます 。 Cmd/Ctrl + Shift + P を押してパレットを開き、「styler」と入力すると、styler が提供するすべてのショートカットが表示されます 。 その結果を Figure 4.1 に示します 。

【Geminiの補足】: 文頭の「butitsuremakesthingseasiertoread」は、英語の “but it sure makes things easier to read”（しかし、それは確実に物事を読みやすくしてくれます）という一文からスペースをすべて取り除いたものです。スペースや句読点がないとどれほど読みにくいかを、身をもって体験させるためのジョークになっています。

この章のコード例では、tidyverse パッケージと nycflights13 パッケージを使用します。

library(tidyverse)
library(nycflights13)
#> Warning: package 'nycflights13' was built under R version 4.5.3

4.1 名前

名前については、Section 2.3 で簡単に触れました 。 変数名（<- や mutate() で作成されるもの）には、小文字、数字、および _（アンダースコア）のみを使用してください 。 名前の中の単語の区切りには _ を使用します 。

# 推奨:
short_flights <- flights |> filter(air_time < 60)

# 回避:
SHORTFLIGHTS <- flights |> filter(air_time < 60)

一般的な経験則として、入力が速い簡潔な名前よりも、理解しやすい長くて説明的な名前を好む方が良いでしょう 。 コードを書く際、短い名前で節約できる時間はごくわずかです（特にオートコンプリートが入力の完成を助けてくれるため）。しかし、古いコードに戻ったときに暗号のような略語を解読せざるを得なくなると、非常に時間がかかります 。 関連するものの名前が複数ある場合は、できる限り一貫性を保つようにしてください 。 以前の慣習を忘れて不一致が生じることはよくあるので、後で戻って名前を変更することになっても気にする必要はありません 。 一般的に、一つのテーマに基づいたバリエーションの変数が複数ある場合は、共通の接尾辞（サフィックス）ではなく共通の接頭辞（プレフィックス）を付ける方が、オートコンプリートが変数の先頭で最もよく機能するため便利です 。

4.2 スペース

^ を除く数学演算子（+, -, ==, <, …）の両側、および代入演算子（<-）の周りにはスペースを入れてください 。

# 推奨
z <- (a + b)^2 / d

# 回避
z<-( a + b ) ^ 2/d

通常の関数呼び出しでは、括弧の内側や外側にスペースを入れないでください 。 標準的な英語と同様に、カンマの後には常にスペースを入れます 。

# 推奨
mean(x, na.rm = TRUE)

# 回避
mean (x ,na.rm=TRUE)

配置（アライメント）を改善するために追加のスペースを入れるのは問題ありません 。 たとえば、mutate() で複数の変数を作成する場合、= の位置が揃うようにスペースを追加することがあります 。 これにより、コードをざっと読みやすくなります 。

flights |>
  mutate(
    speed      = distance / air_time,
    dep_hour   = dep_time %/% 100,
    dep_minute = dep_time %%  100
  )

4.3 パイプ

|> の前には常にスペースを入れ、通常は行の最後に置くようにしてください 。 これにより、新しいステップの追加、既存のステップの入れ替え、ステップ内の要素の変更が容易になり、左側の「動詞」を眺めるだけで全体の俯瞰（10,000 ft view）が可能になります 。

# 推奨
flights |>
  filter(!is.na(arr_delay), !is.na(tailnum)) |>
  count(dest)

# 回避
flights|>filter(!is.na(arr_delay), !is.na(tailnum))|>count(dest)

【Geminiの補足】: 「10,000 ft view（1万フィートの視点）」とは、細かい部分にとらわれず、高いところから全体を眺める「鳥瞰図」や「全体像」を意味するビジネスやデータサイエンスの比喩表現です。

パイプでつなぐ先の関数に名前付き引数（mutate() や summarize() など）がある場合は、各引数を新しい行に配置します 。 関数に名前付き引数がない場合（select() や filter() など）は、一行に収まらない限り一行にまとめ、収まらない場合は各引数を独自の行に配置してください 。

# 推奨
flights |>
  group_by(tailnum) |>
  summarize(
    delay = mean(arr_delay, na.rm = TRUE),
    n = n()
  )

# 回避
flights |>
  group_by(
    tailnum
  ) |>
  summarize(delay = mean(arr_delay, na.rm = TRUE), n = n())

パイプラインの最初のステップ以降、各行を 2 つのスペースでインデントします 。 RStudio では、|> の後に改行を入れると、自動的にスペースが挿入されます 。 各引数を独自の行に配置する場合は、さらに 2 つのスペースでインデントしてください 。 最後の閉じ括弧 ) は、関数名と同じ水平位置になるよう、インデントせずに独自の行に配置してください 。

# 推奨
flights |>
  group_by(tailnum) |>
  summarize(
    delay = mean(arr_delay, na.rm = TRUE),
    n = n()
  )

# 回避
flights|>
  group_by(tailnum) |>
  summarize(
             delay = mean(arr_delay, na.rm = TRUE),
             n = n()
           )

# 回避
flights|>
  group_by(tailnum) |>
  summarize(
  delay = mean(arr_delay, na.rm = TRUE),
  n
  = n()
  )

パイプラインが一行にすっきり収まる場合は、これらのルールの一部を無視しても構いません 。 しかし私たちの共通の経験では、短いコード断片が次第に長くなっていくことはよくあるため、最初から垂直方向のスペースを十分に確保しておくことで、長期的には時間を節約できることがほとんどです 。

# これは一行にコンパクトに収まります
df |> mutate(y = x + 1)

# こちらは 4 倍の行数を使いますが、将来的に
# 変数やステップを増やすのが簡単です
df |>
  mutate(
    y = x + 1
  )

最後に、10〜15 行を超えるような非常に長いパイプラインの作成には注意してください 。 それらをより小さなサブタスクに分割し、それぞれのタスクに説明的な名前を付けてください 。 名前を付けることで、読者は何が起きているのかを把握しやすくなり、中間結果が期待通りであるかの確認も容易になります 。 データ構造を根本的に変更する場合（ピボットや要約の後など）は、常に説明的な名前を付けるべきです 。 最初から完璧にできるとは思わないでください 。 良い名前が付けられる中間状態がある場合は、長いパイプラインを分割しましょう 。

4.4 ggplot2

パイプに適用されるのと同じ基本ルールが ggplot2 にも適用されます。単に + を |> と同じように扱ってください 。

flights |>
  group_by(month) |>
  summarize(
    delay = mean(arr_delay, na.rm = TRUE)
  ) |>
  ggplot(aes(x = month, y = delay)) +
  geom_point() +
  geom_line()

同様に、関数のすべての引数が一行に収まらない場合は、各引数を独自の行に配置します。

flights |>
  group_by(dest) |>
  summarize(
    distance = mean(distance),
    speed = mean(distance / air_time, na.rm = TRUE)
  ) |>
  ggplot(aes(x = distance, y = speed)) +
  geom_smooth(
    method = "loess",
    span = 0.5,
    se = FALSE,
    color = "white",
    linewidth = 4
  ) +
  geom_point()

|> から + への切り替えに注意してください 。 この切り替えが不要であれば良かったのですが、残念ながら ggplot2 はパイプが発見される前に作成されました 。

4.7 まとめ

この章では、コードスタイルの最も重要な原則を学びました 。 最初はこれらが恣意的なルールの集まりに感じるかもしれませんが（実際そうなのです！）、時間の経過とともに、より多くのコードを書き、より多くの人とコードを共有するようになれば、一貫したスタイルの重要性がわかるでしょう 。 styler パッケージを忘れないでください。スタイルの悪いコードの質を素早く向上させる素晴らしい方法です 。 次の章では、データサイエンスのツールに戻り、整然データ（tidy data）について学びます 。 整然データは、tidyverse 全体で使用されるデータフレームの整理の一貫した方法です 。 この一貫性はあなたの生活を楽にしてくれます。一度整然データになれば、大多数の tidyverse 関数でそのまま動作するからです 。 もちろん、現実はそれほど甘くありません。野良（in the wild）で遭遇するほとんどのデータセットは、最初から整然としているわけではありません 。 そのため、tidyr パッケージを使用して、整然としていないデータを整然とした形式にする方法も学びます 。
