21  データベース

21.1 はじめに

膨大な量のデータがデータベースに保存されているため、それらにアクセスする方法を知っておくことは不可欠です。 時には誰かに頼んで .csv ファイルにスナップショットをダウンロードしてもらうこともできますが、これではすぐに立ち行かなくなります。変更が必要になるたびに、他の人とコミュニケーションを取らなければならないからです。 必要なときに、必要なデータを取得するために、データベースに直接アクセスできるようになりたいと思うはずです。 この章では、まず DBI パッケージの基本を学びます。これを使用してデータベースに接続し、SQL[^databases-1] クエリでデータを取得する方法を学習します。 SQL（structured query language の略）はデータベースの共通言語であり、すべてのデータサイエンティストが学ぶべき重要な言語です。 とはいえ、いきなり SQL から始めるのではなく、代わりに dbplyr を教えます。これは、dplyr のコードを SQL に翻訳してくれるツールです。 これを、SQL の最も重要な機能のいくつかを紹介するための手段として利用します。 この章を読み終える頃には SQL の達人になっているわけではありませんが、最も重要な構成要素を特定し、それらが何をするのかを理解できるようになります。 [^databases-1]: SQL は「エス・キュー・エル」または「シークェル」と発音されます。

21.1.1 前提条件

この章では、DBI と dbplyr を紹介します。 DBI は、データベースに接続して SQL を実行する低レベルのインターフェースです。 dbplyr は、dplyr のコードを SQL クエリに翻訳し、DBI を使用して実行する高レベルのインターフェースです。

library(DBI)
library(dbplyr)
library(tidyverse)

21.2 データベースの基本

最も単純なレベルでは、データベースをデータフレームの集合体、データベース用語でいうところのテーブルの集まりと考えることができます。 データフレームと同様に、データベースのテーブルは名前付きの列の集まりであり、列内のすべての値は同じ型です。 データフレームとデータベースのテーブルには、大きく分けて 3 つの違いがあります。

  • データベースのテーブルはディスク上に保存され、いくらでも大きくすることができます。 データフレームはメモリ内に保存されるため、根本的な制限があります（ただし、その制限は多くの問題に対して依然として十分に大きいです）。

  • データベースのテーブルには、ほとんどの場合インデックス（索引）があります。 本の索引と同じように、データベースのインデックスを使用すると、すべての行を確認することなく、目的の行をすばやく見つけることができます。 データフレームや tibble にはインデックスがありませんが、data.table にはあり、それが高速である理由の一つとなっています。

  • ほとんどの古典的なデータベースは、既存のデータの分析ではなく、データの迅速な収集に最適化されています。 これらのデータベースは、R のように列ごとにデータを保存するのではなく、行ごとに保存されるため、行指向と呼ばれます。 最近では、既存のデータの分析を大幅に高速化する列指向データベースの開発が盛んに行われています。 データベースはデータベース管理システム（略して DBMS）によって運用されており、主に 3 つの形式があります。

  • クライアント・サーバー型 DBMS は強力な中央サーバー上で動作し、自分のコンピュータ（クライアント）から接続します。 組織内の複数の人でデータを共有するのに適しています。 一般的なクライアント・サーバー型 DBMS には、PostgreSQL、MariaDB、SQL Server、Oracle などがあります。

  • クラウド型 DBMS（Snowflake、Amazon の RedShift、Google の BigQuery など）は、クライアント・サーバー型 DBMS と似ていますが、クラウド上で動作します。 つまり、極めて大規模なデータセットを容易に扱うことができ、必要に応じて計算リソースを自動的に増強できます RedShift。

  • インプロセス型 DBMS（SQLite や duckdb など）は、すべて自分のコンピュータ上で動作します。 あなたが主なユーザーである場合に、大規模なデータセットを扱うのに適しています。

21.3 データベースへの接続

R からデータベースに接続するには、一組のパッケージを使用します。

  • DBI (database interface) は常に使用します。これは、データベースへの接続、データのアップロード、SQL クエリの実行などのための汎用的な関数群を提供するためです。

  • 接続先の DBMS に合わせたパッケージも使用します。 このパッケージは、汎用的な DBI コマンドを、特定の DBMS に必要な具体的なコマンドに変換します。 通常、DBMS ごとに 1 つのパッケージがあります。例えば、PostgreSQL 用の RPostgres や MySQL 用の RMariaDB などです。 特定の DBMS 用のパッケージが見つからない場合は、通常 odbc パッケージを使用できます。 これは、多くの DBMS でサポートされている ODBC プロトコルを使用します。 odbc は、ODBC ドライバーをインストールし、odbc パッケージにその場所を教える必要があるため、少し設定が必要です。 具体的には、DBI::dbConnect() を使用してデータベース接続を作成します。 最初の引数で DBMS を選択し1、2 番目以降の引数で接続方法（場所やアクセスに必要な資格情報など）を指定します。 次のコードは、いくつかの典型的な例を示しています。

con <- DBI::dbConnect(
  RMariaDB::MariaDB(),
  username = "foo"
)
con <- DBI::dbConnect(
  RPostgres::Postgres(),
  hostname = "databases.mycompany.com",
  port = 1234
)

接続の詳細は DBMS ごとに大きく異なるため、残念ながらここではすべての詳細を網羅することはできません。 つまり、自分自身で少し調べる必要があります。 通常は、チーム内の他のデータサイエンティストに尋ねるか、DBA（database administrator：データベース管理者）に相談するとよいでしょう。 最初の設定を正しく行うには、少し調整（そしておそらく Google 検索）が必要になることがよくありますが、通常は一度行うだけで済みます。

21.3.1 本書では

本書のためにクライアント・サーバー型やクラウド型の DBMS をセットアップするのは大変なので、代わりに R パッケージ内で完全に動作するインプロセス型 DBMS である duckdb を使用します。 DBI の魔法のおかげで、duckdb を使う場合と他の DBMS を使う場合の違いは、データベースへの接続方法だけです。 そのため、このコードを簡単に実行できるだけでなく、学んだことを他の場所で簡単に応用できるので、学習に最適です。 duckdb への接続は、デフォルトで R を終了すると削除される一時的なデータベースが作成されるため、特に簡単です。 これは、R を再起動するたびにクリーンな状態から始められることが保証されるため、学習には最適です。

con <- DBI::dbConnect(duckdb::duckdb())

duckdb は、データサイエンティストのニーズに合わせて設計された高性能なデータベースです。 ここでは始めやすさから使用していますが、数ギガバイトのデータを非常に高速に処理する能力も備えています。 実際のデータ分析プロジェクトで duckdb を使用する場合は、永続的なデータベースを作成するために dbdir 引数を指定し、duckdb に保存場所を教える必要があります。 プロジェクト（Chapter 6）を使用している場合は、現在のプロジェクトの duckdb ディレクトリに保存するのが妥当です。

con <- DBI::dbConnect(duckdb::duckdb(), dbdir = "duckdb")

21.3.2 データのロード

これは新しいデータベースなので、まずデータを追加する必要があります。 ここでは DBI::dbWriteTable() を使用して、ggplot2 から mpg と diamonds データセットを追加します。 dbWriteTable() の最も単純な使い方は、データベース接続、データベース内に作成するテーブルの名前、およびデータのデータフレームという 3 つの引数を必要とします。

dbWriteTable(con, "mpg", ggplot2::mpg)
dbWriteTable(con, "diamonds", ggplot2::diamonds)

実際のプロジェクトで duckdb を使用する場合は、duckdb_read_csv() と duckdb_register_arrow() について学ぶことを強くお勧めします。 これらは、最初に R に読み込むことなく、データを duckdb に直接迅速にロードするための強力でパフォーマンスの高い方法を提供します。 Section 26.4.1 では、複数のファイルをデータベースにロードするための便利な手法も紹介します。

21.3.3 DBI の基本

他のいくつかの DBI 関数を使用して、データが正しくロードされたかを確認できます。dbListTables() はデータベース内のすべてのテーブルをリストし2、dbReadTable() はテーブルの内容を取得します。

dbListTables(con)
#> [1] "diamonds" "mpg"

con |>

sql <- "

"

21.4 dbplyr の基本

データベースに接続してデータをロードできたので、dbplyr について学び始めましょう。 dbplyr は dplyr のバックエンドです。つまり、引き続き dplyr のコードを書き続けますが、バックエンドがそれを異なる方法で実行します。 この場合、dbplyr は SQL に翻訳します。他のバックエンドには、data.table に翻訳する dtplyr や、複数のコアでコードを実行する multidplyr などがあります。 dbplyr を使用するには、まず tbl() を使用してデータベースのテーブルを表すオブジェクトを作成する必要があります。

diamonds_db <- tbl(con, "diamonds")
diamonds_db
#> # Source:   table<diamonds> [?? x 10]
#> # Database: DuckDB 1.5.0 [777mi@Windows 10 x64:R 4.5.1/:memory:]
#>   carat cut       color clarity depth table price     x     y     z
#>   <dbl> <fct>     <fct> <fct>   <dbl> <dbl> <int> <dbl> <dbl> <dbl>
#> 1  0.23 Ideal     E     SI2      61.5    55   326  3.95  3.98  2.43
#> 2  0.21 Premium   E     SI1      59.8    61   326  3.89  3.84  2.31
#> 3  0.23 Good      E     VS1      56.9    65   327  4.05  4.07  2.31
#> 4  0.29 Premium   I     VS2      62.4    58   334  4.2   4.23  2.63
#> 5  0.31 Good      J     SI2      63.3    58   335  4.34  4.35  2.75
#> 6  0.24 Very Good J     VVS2     62.8    57   336  3.94  3.96  2.48
#> # ℹ more rows

データベースとやり取りする一般的な方法は、他に 2 つあります。 1 つ目は、多くの企業のデータベースは非常に大規模なため、すべてのテーブルを整理しておくための階層が必要になる場合です。 その場合、関心のあるテーブルを選択するために、スキーマ、あるいはカタログとスキーマを指定する必要があるかもしれません。

diamonds_db <- tbl(con, in_schema("sales", "diamonds"))
diamonds_db <- tbl(con, in_catalog("north_america", "sales", "diamonds"))

もう 1 つは、独自の SQL クエリを出発点として使用したい場合です。

diamonds_db <- tbl(con, sql("SELECT * FROM diamonds"))

このオブジェクトは遅延型（lazy）です。 これに対して dplyr の動詞を使用しても、dplyr は何もしません。単に実行したい一連の操作を記録し、必要になったときにのみ実行します。 例えば、次のパイプラインを見てみましょう。

big_diamonds_db <- diamonds_db |>
  filter(price > 15000) |>
  select(carat:clarity, price)

big_diamonds_db
#> # Source:   SQL [?? x 5]
#> # Database: DuckDB 1.5.0 [777mi@Windows 10 x64:R 4.5.1/:memory:]
#>   carat cut       color clarity price
#>   <dbl> <fct>     <fct> <fct>   <int>
#> 1  1.54 Premium   E     VS2     15002
#> 2  1.19 Ideal     F     VVS1    15005
#> 3  2.1  Premium   I     SI1     15007
#> 4  1.69 Ideal     D     SI1     15011
#> 5  1.5  Very Good G     VVS2    15013
#> 6  1.73 Very Good G     VS1     15014
#> # ℹ more rows

このオブジェクトがデータベースクエリを表していることは、上部に DBMS 名が表示されていることからわかります。また、列の数は示されていますが、通常、行の数はわかりません。 これは、総行数を確認するには通常クエリ全体を実行する必要があり、それを避けようとしているためです。 dplyr 関数の show_query() を使うと、生成された SQL コードを確認できます。 dplyr を知っているなら、これは SQL を学ぶのに最適な方法です。 dplyr のコードを書き、dbplyr に SQL へ翻訳させ、2 つの言語がどのように対応しているかを確認してみてください。

big_diamonds_db |>
  show_query()
#> <SQL>
#> SELECT carat, cut, color, clarity, price
#> FROM diamonds
#> WHERE (price > 15000.0)

すべてのデータを R に戻すには、collect() を呼び出します。 舞台裏では、これが SQL を生成し、dbGetQuery() を呼び出してデータを取得し、結果を tibble に変換します。

big_diamonds <- big_diamonds_db |>
  collect()
big_diamonds
#> # A tibble: 1,655 × 5
#>   carat cut       color clarity price
#>   <dbl> <fct>     <fct> <fct>   <int>
#> 1  1.54 Premium   E     VS2     15002
#> 2  1.19 Ideal     F     VVS1    15005
#> 3  2.1  Premium   I     SI1     15007
#> 4  1.69 Ideal     D     SI1     15011
#> 5  1.5  Very Good G     VVS2    15013
#> 6  1.73 Very Good G     VS1     15014
#> # ℹ 1,649 more rows

通常、dbplyr を使用してデータベースから必要なデータを選択し、後述する翻訳機能を使用して基本的なフィルタリングや集計を行います。 そして、R 独自の関数でデータを分析する準備ができたら、データを collect() してメモリ内の tibble を取得し、純粋な R のコードで作業を続けます。

21.5 SQL

この章の残りの部分では、dbplyr を通じて SQL を少し学びます。 かなり型破りな SQL の入門編ですが、基本を素早く習得できることを願っています。 幸いなことに、dplyr を理解していれば、多くの概念が同じであるため、SQL をすぐに習得できる素晴らしい土台ができています。 nycflights13 パッケージでお馴染みの flights と planes を使って、dplyr と SQL の関係を探っていきます。 これらのデータセットを学習用データベースに取り込むのは簡単です。dbplyr には nycflights13 からデータベースにテーブルをコピーする関数が用意されているからです。

dbplyr::copy_nycflights13(con)
#> Creating table: airlines
#> Creating table: airports
#> Creating table: flights
#> Creating table: planes
#> Creating table: weather
flights <- tbl(con, "flights")
planes <- tbl(con, "planes")

21.5.1 SQL の基本

SQL の最上位コンポーネントは文（ステートメント）と呼ばれます。 一般的な文には、新しいテーブルを定義する CREATE、データを追加する INSERT、データを取得する SELECT などがあります。 ここでは、データサイエンティストがほぼ排他的に使用する SELECT 文（クエリとも呼ばれます）に焦点を当てます。 クエリは句（クローズ）で構成されます。 重要な句は SELECT、FROM、WHERE、ORDER BY、GROUP BY の 5 つです。 すべてのクエリには SELECT3 句と FROM4 句が必要です。最も単純なクエリは SELECT * FROM table で、指定したテーブルからすべての列を選択します。 これは、加工されていないテーブルに対して dbplyr が生成するものです。

flights |> show_query()
#> <SQL>
#> SELECT *
#> FROM flights
planes |> show_query()
#> <SQL>
#> SELECT *
#> FROM planes

WHERE と ORDER BY は、どの行を含めるか、およびそれらをどのように並べるかを制御します。

flights |>
  filter(dest == "IAH") |>
  arrange(dep_delay) |>
  show_query()
#> <SQL>
#> SELECT flights.*
#> FROM flights
#> WHERE (dest = 'IAH')
#> ORDER BY dep_delay

GROUP BY はクエリを集計に変換し、集約処理を行わせます。

flights |>
  group_by(dest) |>
  summarize(dep_delay = mean(dep_delay, na.rm = TRUE)) |>
  show_query()
#> <SQL>
#> SELECT dest, AVG(dep_delay) AS dep_delay
#> FROM flights
#> GROUP BY dest

dplyr の動詞と SELECT 句の間には、2 つの重要な違いがあります。

  • SQL では大文字と小文字は関係ありません。select、SELECT、さらには SeLeCt と書くこともできます。 本書では、テーブル名や変数名と区別するために、SQL のキーワードを大文字で書くという一般的な慣習に従います。
  • SQL では順序が重要です。句は常に SELECT、FROM、WHERE、GROUP BY、ORDER BY の順序で書かなければなりません。 紛らわしいことに、この順序は句が実際に評価される順序（まず FROM、次に WHERE、GROUP BY、SELECT、そして ORDER BY）とは一致しません。 次のセクションでは、各句をさらに詳しく見ていきます。

SQL は標準規格ではありますが、非常に複雑であり、完全に準拠しているデータベースはありません。 本書で焦点を当てる主要なコンポーネントは DBMS 間で非常によく似ていますが、多くの細かなバリエーションが存在します。 幸いなことに、dbplyr はこの問題に対処するように設計されており、データベースごとに異なる翻訳を生成します。 完璧ではありませんが、継続的に改善されています。もし問題が発生した場合は、GitHub で問題を報告して、私たちの改善を助けてください。

21.5.2 SELECT

SELECT 句はクエリの主力であり、select()、mutate()、rename()、relocate()、そして次のセクションで学ぶように summarize() と同じ役割を果たします。 select()、rename()、relocate() は、列がどこに表示されるか（または表示されないか）とその名前にのみ影響するため、SELECT への非常に直接的な翻訳を持ちます。

planes |>
  select(tailnum, type, manufacturer, model, year) |>
  show_query()
#> <SQL>
#> SELECT tailnum, "type", manufacturer, model, "year"
#> FROM planes

planes |>
  select(tailnum, type, manufacturer, model, year) |>
  rename(year_built = year) |>
  show_query()
#> <SQL>
#> SELECT tailnum, "type", manufacturer, model, "year" AS year_built
#> FROM planes

planes |>
  select(tailnum, type, manufacturer, model, year) |>
  relocate(manufacturer, model, .before = type) |>
  show_query()
#> <SQL>
#> SELECT tailnum, manufacturer, model, "type", "year"
#> FROM planes

この例は、SQL がどのようにリネームを行うかも示しています。 SQL の用語では、リネームはエイリアシング（別名定義）と呼ばれ、AS を使って行われます。 mutate() とは異なり、古い名前が左側に、新しい名前が右側にあることに注意してください。 ::: callout-note 上記の例で "year" や "type" が二重引用符で囲まれていることに注目してください。 これは、これらが duckdb における予約語であるため、dbplyr が列名/テーブル名と SQL 演算子の混同を避けるために引用符で囲んでいるのです。 他のデータベースで作業する場合、duckdb のようにどの単語が予約語であるかを把握しているクライアントパッケージは一握りしかないため、安全のためにすべての変数名が引用符で囲まれているのを目にすることが多いでしょう。

SELECT "tailnum", "type", "manufacturer", "model", "year"
FROM "planes"

データベースシステムによっては、引用符の代わりにバッククォートを使用するものもあります。

SELECT `tailnum`, `type`, `manufacturer`, `model`, `year`
FROM `planes`

:::

mutate() の翻訳も同様に単純です。各変数が SELECT 内の新しい式になります。

flights |>
  mutate(
    speed = distance / (air_time / 60)
  ) |>
  show_query()
#> <SQL>
#> SELECT flights.*, distance / (air_time / 60.0) AS speed
#> FROM flights

個々の構成要素（/ など）の翻訳については、Section 21.6 で詳しく説明します。

21.5.3 FROM

FROM 句はデータソースを定義します。 今のところは単一のテーブルしか使っていないので、あまり面白くありません。 結合（join）関数を扱うようになると、より複雑な例を見ることになります。

21.5.4 GROUP BY

group_by() は GROUP BY5 句に、summarize() は SELECT 句に翻訳されます。

diamonds_db |>
  group_by(cut) |>
  summarize(
    n = n(),
    avg_price = mean(price, na.rm = TRUE)
  ) |>
  show_query()
#> <SQL>
#> SELECT cut, COUNT(*) AS n, AVG(price) AS avg_price
#> FROM diamonds
#> GROUP BY cut

n() と mean() の翻訳で何が起きているかについては、Section 21.6 で詳しく説明します。

21.5.5 WHERE

filter() は WHERE 句に翻訳されます。

flights |>
  filter(dest == "IAH" | dest == "HOU") |>
  show_query()
#> <SQL>
#> SELECT flights.*
#> FROM flights
#> WHERE (dest = 'IAH' OR dest = 'HOU')

flights |>
  filter(arr_delay > 0 & arr_delay < 20) |>
  show_query()
#> <SQL>
#> SELECT flights.*
#> FROM flights
#> WHERE (arr_delay > 0.0 AND arr_delay < 20.0)

ここで注目すべき重要な点がいくつかあります。

  • | は OR に、& は AND になります。
  • SQL では比較に == ではなく = を使用します。 SQL には代入がないため、混乱の余地はありません。
  • SQL では文字列に "" ではなく '' のみを使用します。 SQL では、"" は R の `` のように変数を特定するために使用されます。 もう一つの便利な SQL 演算子は IN です。これは R の %in% に非常によく似ています。
flights |>
  filter(dest %in% c("IAH", "HOU")) |>
  show_query()
#> <SQL>
#> SELECT flights.*
#> FROM flights
#> WHERE (dest IN ('IAH', 'HOU'))

SQL では NA の代わりに NULL を使用します。 NULL は NA と似たような挙動をします。 主な違いは、比較や算術演算においては「伝染」しますが、集計時には黙って除外される点です。 dbplyr は、この挙動に初めて遭遇したときに注意を促してくれます。

flights |>
  group_by(dest) |>
  summarize(delay = mean(arr_delay))
#> Warning: Missing values are always removed in SQL aggregation functions.
#> Use `na.rm = TRUE` to silence this warning
#> This warning is displayed once every 8 hours.
#> # Source:   SQL [?? x 2]
#> # Database: DuckDB 1.5.0 [777mi@Windows 10 x64:R 4.5.1/:memory:]
#>   dest   delay
#>   <chr>  <dbl>
#> 1 CLT    7.36 
#> 2 MDW   12.4  
#> 3 SDF   12.7  
#> 4 LAS    0.258
#> 5 IAH    4.24 
#> 6 CAK   19.7  
#> # ℹ more rows

NULL の仕組みについて詳しく知りたい場合は、Markus Winand による “The Three-Valued Logic of SQL” が参考になるでしょう。 一般的に、R で NA に対して使う関数を使用して NULL を扱うことができます。

flights |>
  filter(!is.na(dep_delay)) |>
  show_query()
#> <SQL>
#> SELECT flights.*
#> FROM flights
#> WHERE (NOT((dep_delay IS NULL)))

この SQL クエリは、dbplyr の欠点の一つを浮き彫りにしています。SQL としては正しいのですが、手書きするほどシンプルではありません。 この場合、括弧を外して、より読みやすい特別な演算子を使用することもできます。

WHERE "dep_delay" IS NOT NULL

summarize を使用して作成した変数を filter() すると、dbplyr は WHERE 句ではなく HAVING 句を生成することに注意してください。 これは SQL の特異性の一つです。WHERE は SELECT や GROUP BY の前に評価されるため、SQL にはそれらの後に評価される別の句が必要なのです。

diamonds_db |>
  group_by(cut) |>
  summarize(n = n()) |>
  filter(n > 100) |>
  show_query()
#> <SQL>
#> SELECT cut, COUNT(*) AS n
#> FROM diamonds
#> GROUP BY cut
#> HAVING (COUNT(*) > 100.0)

21.5.6 ORDER BY

行の並び替えは、arrange() から ORDER BY 句へと直接翻訳されます。

flights |>
  arrange(year, month, day, desc(dep_delay)) |>
  show_query()
#> <SQL>
#> SELECT flights.*
#> FROM flights
#> ORDER BY "year", "month", "day", dep_delay DESC

desc() が DESC に翻訳されていることに注目してください。これも、SQL から直接名前が採用された多くの dplyr 関数の一つです。

21.5.7 サブクエリ

dplyr パイプラインを単一の SELECT 文に翻訳することができず、サブクエリを使用しなければならない場合があります。 サブクエリとは、通常のテーブルの代わりに、FROM 句でデータソースとして使用されるクエリのことです。 dbplyr は通常、SQL の制限を回避するためにサブクエリを使用します。 例えば、SELECT 句内の式では、作成したばかりの列を参照することができません。 つまり、次のような（愚直な）dplyr パイプラインは 2 つのステップで実行される必要があります。最初の（内部）クエリが year1 を計算し、次に 2 番目の（外部）クエリが year2 を計算できるようにします。

flights |>
  mutate(
    year1 = year + 1,
    year2 = year1 + 1
  ) |>
  show_query()
#> <SQL>
#> SELECT q01.*, year1 + 1.0 AS year2
#> FROM (
#>   SELECT flights.*, "year" + 1.0 AS year1
#>   FROM flights
#> ) q01

作成したばかりの変数を filter() しようとした場合にも、これを目にするでしょう。 WHERE は SELECT よりも後に書かれますが、評価されるのは先であることを思い出してください。そのため、この（愚直な）例ではサブクエリが必要になります。

flights |>
  mutate(year1 = year + 1) |>
  filter(year1 == 2014) |>
  show_query()
#> <SQL>
#> SELECT q01.*
#> FROM (
#>   SELECT flights.*, "year" + 1.0 AS year1
#>   FROM flights
#> ) q01
#> WHERE (year1 = 2014.0)

dbplyr がまだその翻訳を最適化する方法を知らないため、必要のない場所でサブクエリを作成することもあります。 dbplyr が時間の経過とともに改善されるにつれて、このようなケースは少なくなりますが、完全になくなることはないでしょう。

21.5.8 結合 (Joins)

dplyr の結合に慣れているなら、SQL の結合も非常によく似ています。 以下に簡単な例を示します。

flights |>
  left_join(planes |> rename(year_built = year), join_by(tailnum)) |>
  show_query()
#> <SQL>
#> SELECT
#>   flights.*,
#>   planes."year" AS year_built,
#>   "type",
#>   manufacturer,
#>   model,
#>   engines,
#>   seats,
#>   speed,
#>   engine
#> FROM flights
#> LEFT JOIN planes
#>   ON (flights.tailnum = planes.tailnum)

ここで注目すべき主な点は構文です。SQL の結合は、FROM 句のサブ句を使用して追加のテーブルを取り込み、ON を使用してテーブルがどのように関連しているかを定義します。 dplyr の関数名は SQL と密接に関連しているため、inner_join()、right_join()、full_join() に相当する SQL を容易に推測できます。

SELECT flights.*, "type", manufacturer, model, engines, seats, speed
FROM flights
INNER JOIN planes ON (flights.tailnum = planes.tailnum)

SELECT flights.*, "type", manufacturer, model, engines, seats, speed
FROM flights
RIGHT JOIN planes ON (flights.tailnum = planes.tailnum)

SELECT flights.*, "type", manufacturer, model, engines, seats, speed
FROM flights
FULL JOIN planes ON (flights.tailnum = planes.tailnum)

データベースのデータを扱う場合、多くの結合が必要になるでしょう。 それは、データベースのテーブルが高度に正規化された形式で保存されていることが多いためです。各「事実」は 1 か所に保存されており、分析のために完全なデータセットを保持するには、主キーと外部キーで接続された複雑なテーブルネットワークをナビゲートする必要があります。 このような状況に遭遇した場合、Tobias Schieferdecker、Kirill Müller、Darko Bergant による dm パッケージが救世主となります。 これは、DBA が提供することが多い制約を使用してテーブル間の接続を自動的に判断し、何が起きているかを確認できるように接続を可視化し、あるテーブルを別のテーブルに接続するために必要な結合を生成してくれます。

21.5.9 その他の動詞

dbplyr は distinct()、slice_*()、intersect() などの他の動詞や、pivot_longer()、pivot_wider() などの tidyr 関数の翻訳も増やしています。 現在利用可能な全セットを確認する最も簡単な方法は、dbplyr のウェブサイト https://dbplyr.tidyverse.org/reference/ を訪れることです。

21.5.10 練習問題

  1. distinct() は何に翻訳されますか？ head() はどうですか？
  2. 次の各 SQL クエリが何をするか説明し、dbplyr を使用してそれらを再現してみてください。
    SELECT * FROM flights
    WHERE dep_delay < arr_delay

    SELECT *, distance / (air_time / 60) AS speed

21.6 関数の翻訳

これまでは、dplyr の動詞がクエリの各句にどのように翻訳されるかという全体像に焦点を当ててきました。 ここでは少しズームインして、個々の列を操作する R 関数の翻訳について説明します。例えば、summarize() で mean(x) を使用すると何が起こるでしょうか？ 何が起きているかを確認しやすくするために、summarize() または mutate() を実行して生成された SQL を表示する 2 つの補助関数を使用します。 これにより、いくつかのバリエーションを探索し、集計と変換がどのように異なるかを確認しやすくなります。

summarize_query <- function(df, ...) {
  df |>
    summarize(...) |>
    show_query()
}
mutate_query <- function(df, ...) {
  df |>
    mutate(..., .keep = "none") |>
    show_query()
}

まずは集計から見ていきましょう！ 以下のコードを見ると、mean() のような一部の集計関数は比較的単純な翻訳になりますが、median() のような他の関数ははるかに複雑であることがわかります。 統計では一般的でも、データベースではあまり一般的ではない操作ほど、複雑になる傾向があります。

flights |>
  group_by(year, month, day) |>
  summarize_query(
    mean = mean(arr_delay, na.rm = TRUE),
    median = median(arr_delay, na.rm = TRUE)
  )
#> `summarise()` has grouped output by "year" and "month". You can override
#> using the `.groups` argument.
#> <SQL>
#> SELECT
#>   "year",
#>   "month",
#>   "day",
#>   AVG(arr_delay) AS mean,
#>   MEDIAN(arr_delay) AS median
#> FROM flights
#> GROUP BY "year", "month", "day"

集計関数を mutate() の中で使用すると、いわゆるウィンドウ（window）関数に変換されるため、翻訳がより複雑になります。 SQL では、通常の集約関数の後に OVER を追加することで、ウィンドウ関数に変換します。

flights |>
  group_by(year, month, day) |>
  mutate_query(
    mean = mean(arr_delay, na.rm = TRUE),
  )
#> <SQL>
#> SELECT
#>   "year",
#>   "month",
#>   "day",
#>   AVG(arr_delay) OVER (PARTITION BY "year", "month", "day") AS mean
#> FROM flights

SQL では GROUP BY 句は集計専用に使用されるため、ここではグルーピングが GROUP BY 句から OVER に移動していることがわかります。 ウィンドウ関数には、前方または後方を確認するすべての関数が含まれます。例えば、それぞれ「前」または「次」の値を確認する lead() や lag() などです。

flights |>
  group_by(dest) |>
  arrange(time_hour) |>
  mutate_query(
    lead = lead(arr_delay),
    lag = lag(arr_delay)
  )
#> <SQL>
#> SELECT
#>   dest,
#>   LEAD(arr_delay, 1, NULL) OVER (PARTITION BY dest ORDER BY time_hour) AS lead,
#>   LAG(arr_delay, 1, NULL) OVER (PARTITION BY dest ORDER BY time_hour) AS lag
#> FROM flights
#> ORDER BY time_hour

ここでは、SQL テーブルには固有の順序がないため、データを arrange() しておくことが重要です。 実際、arrange() を使用しないと、実行するたびに行が異なる順序で返される可能性があります。 ウィンドウ関数の場合、順序付けの情報が繰り返されることに注意してください。メインクエリの ORDER BY 句は、ウィンドウ関数に自動的には適用されません。 もう一つの重要な SQL 関数は CASE WHEN です。これは、if_else() や、それが直接着想を与えた dplyr 関数である case_when() の翻訳として使用されます。 以下にいくつかの簡単な例を示します。

flights |>
  mutate_query(
    description = if_else(arr_delay > 0, "delayed", "on-time")
  )
#> <SQL>
#> SELECT CASE WHEN (arr_delay > 0.0) THEN 'delayed' WHEN NOT (arr_delay > 0.0) THEN 'on-time' END AS description
#> FROM flights
flights |>
  mutate_query(
    description =
      case_when(
        arr_delay < -5 ~ "early",
        arr_delay < 5 ~ "on-time",
        arr_delay >= 5 ~ "late"
      )
  )
#> <SQL>
#> SELECT CASE
#> WHEN (arr_delay < -5.0) THEN 'early'
#> WHEN (arr_delay < 5.0) THEN 'on-time'
#> WHEN (arr_delay >= 5.0) THEN 'late'
#> END AS description
#> FROM flights

CASE WHEN は、R から SQL へ直接的な翻訳がない他のいくつかの関数にも使用されます。 その良い例が cut() です。

flights |>
  mutate_query(
    description =  cut(
      arr_delay,
      breaks = c(-Inf, -5, 5, Inf),
      labels = c("early", "on-time", "late")
    )
  )
#> <SQL>
#> SELECT CASE
#> WHEN (arr_delay <= -5.0) THEN 'early'
#> WHEN (arr_delay <= 5.0) THEN 'on-time'
#> WHEN (arr_delay > 5.0) THEN 'late'
#> END AS description
#> FROM flights

dbplyr は一般的な文字列や日付時刻の操作関数も翻訳します。これについては vignette("translation-function", package = "dbplyr") で詳しく学ぶことができます。 dbplyr の翻訳は決して完璧ではなく、まだ翻訳されていない R 関数も多くありますが、dbplyr はほとんどの場合に使用する関数を驚くほどよくカバーしています。

21.7 まとめ

この章では、データベースからデータにアクセスする方法を学びました。 おなじみの dplyr コードを記述し、それを自動的に SQL に翻訳できる dplyr の「バックエンド」である dbplyr に焦点を当てました。 その翻訳を利用して SQL を少し学びました。 SQL はデータを扱うための言語として最も一般的に使用されているため、SQL を少しでも学んでおくことは重要です。SQL を知っていれば、R を使用しない他のデータ関係者とのコミュニケーションが容易になります。

この章を終えて SQL についてさらに詳しく学びたい場合は、次の 2 つをお勧めします。

  • Renée M. P. Teate による SQL for Data Scientists は、データサイエンティストのニーズに合わせて特別に設計された SQL の入門書であり、実際の組織で遭遇するような高度に相互接続されたデータの例が含まれています。
  • Anthony DeBarros による Practical SQL は、データジャーナリスト（説得力のあるストーリーを語ることに特化したデータサイエンティスト）の視点から書かれており、データベースへのデータの取り込みや独自の DBMS の運用について詳しく説明されています。 次の章では、大規模なデータを扱うためのもう一つの dplyr バックエンドである arrow について学びます。 Arrow はディスク上の大きなファイルを扱うように設計されており、データベースを補完する自然な存在です。

  1. 通常、クライアントパッケージから使用する関数はこれだけなので、library() でパッケージ全体を読み込むのではなく、:: を使ってその関数だけを取り出すことをお勧めします。↩︎

  2. 少なくとも、表示権限があるすべてのテーブルです。↩︎

  3. 紛らわしいことに、文脈によって SELECT は文であったり句であったりします。 この混乱を避けるため、通常は SELECT 文の代わりに SELECT クエリという言葉を使います。↩︎

  4. まあ、厳密には SELECT 1+1 のように基本的な計算を行うクエリを書くことができるので、SELECT だけが必要です。 しかし、（常にそうであるように）データを扱いたい場合は、FROM 句も必要になります。↩︎

  5. これは偶然ではありません。dplyr の関数名は SQL 句から着想を得たものです。↩︎
