14  文字列

14.1 はじめに

これまでは詳細をあまり学ばずに、多くの文字列を使用してきました 。ここからは文字列を深く掘り下げ、文字列がどのように機能するかを学び、自由に使える強力な文字列操作ツールをマスターしましょう 。

まず、文字列と文字ベクトル（character vectors）を作成する詳細から始めます 。次に、データから文字列を作成する方法、そしてその逆であるデータから文字列を抽出する方法について深く掘り下げます 。その後、個々の文字を扱うツールについて議論します 。この章の最後では、個々の文字を扱う関数と、英語の感覚が他の言語を扱う際にどのように誤解を招く可能性があるかについての簡単な議論で締めくくります 。文字列については次章でも引き続き扱い、そこでは正規表現の威力についてさらに詳しく学びます 。

14.1.1 事前準備

この章では、tidyverse のコアメンバーである stringr パッケージの関数を使用します 。また、操作するのに楽しい文字列が含まれている babynames データも使用します 。

library(tidyverse)
library(babynames)
#> Warning: package 'babynames' was built under R version 4.5.3

stringr の関数はすべて str_ で始まるため、一目で見分けることができます 。これは RStudio を使用している場合に特に便利で、str_ と入力するとオートコンプリートが起動し、利用可能な関数を思い出すのに役立ちます 。

14.2 文字列の作成

本書の前の部分で、文字列については軽く触れましたが、詳細については議論しませんでした 。まず、文字列はシングルクォート（'）またはダブルクォート（"）のどちらかを使用して作成できます 。両者の間で動作に違いはありません。そのため、一貫性を保つために tidyverse スタイルガイド では、文字列の中に複数の " が含まれている場合を除き、" を使用することを推奨しています 。

string1 <- "これは文字列です"
string2 <- '文字列の中に "引用符" を含めたい場合は、シングルクォートを使用します'

引用符を閉じ忘れると、継続プロンプトである + が表示されます 。

> "これは閉じ引用符のない文字列です
+
+
+ 助けて！文字列の中に閉じ込められました

もしこのようなことが起こり、どの引用符を閉じればよいかわからない場合は、Escape キーを押してキャンセルし、やり直してください 。

14.2.1 エスケープ

文字列の中に文字通りのシングルクォートやダブルクォートを含めるには、\ を使ってそれを「エスケープ」します 。

double_quote <- "\"" # または '"'
single_quote <- '\'' # または "'"

したがって、文字列の中に文字通りのバックスラッシュを含めたい場合は、\\ とエスケープする必要があります 。

backslash <- "\\"

文字列の「印刷された表現」は、文字列そのものとは異なることに注意してください。印刷された表現にはエスケープが表示されます（言い換えれば、文字列を印刷したとき、その出力をコピー＆ペーストしてその文字列を再作成できるようになっています）。文字列の生の（raw）内容を確認するには、str_view() を使用してください 。

x <- c(single_quote, double_quote, backslash)
x
#> [1] "'"  "\"" "\\"
str_view(x)
#> [1] │ '
#> [2] │ "
#> [3] │ \

14.2.2 生の文字列

多くの引用符やバックスラッシュを含む文字列を作成すると、すぐに混乱してしまいます 。この問題を説明するために、double_quote と single_quote 変数を定義したコードブロックの内容を含む文字列を作成してみましょう 。

tricky <- "double_quote <- \"\\\"\" # or '\"'
single_quote <- '\\'' # or \"'\""
str_view(tricky)
#> [1] │ double_quote <- "\"" # or '"'
#>     │ single_quote <- '\'' # or "'"

バックスラッシュが非常に多いですね！ （これは時に leaning toothpick syndrome と呼ばれます。）

【Geminiの補足】: leaning toothpick syndrome（傾いた爪楊枝症候群）とは、バックスラッシュ（\）やスラッシュ（/）を多用してエスケープを繰り返すことで、コードが斜めに傾いた爪楊枝が並んでいるように見え、非常に読みにくくなる状況を指すジョーク交じりの用語です。

このエスケープを排除するために、代わりに生の文字列（raw string）を使用できます 。

tricky <- r"(double_quote <- "\"" # or '"'
single_quote <- '\'' # or "'")"
str_view(tricky)
#> [1] │ double_quote <- "\"" # or '"'
#>     │ single_quote <- '\'' # or "'"

生の文字列は通常 r"( で始まり、) " で終わります 。しかし、もし文字列の中に )" が含まれている場合は、代わりに r"[]" や r"{}" を使用できます。さらにそれだけでは不十分な場合は、任意の数のダッシュを挿入して開始と終了のペアをユニークにすることができます（例：r"--()--", r"---()---" など）。生の文字列は、あらゆるテキストを扱えるほど柔軟です 。

14.2.3 その他の特殊文字

" や '、\ だけでなく、便利な特殊文字がいくつかあります 。最も一般的なのは、改行（new line）の \n とタブ（tab）の \t です 。また、\u や \U で始まる Unicode エスケープを含む文字列を目にすることもあります 。これは、すべてのシステムで動作する英語以外の文字を記述する方法です 。その他の特殊文字の完全なリストは ?Quotes で確認できます 。

x <- c("one\ntwo", "one\ttwo", "\u00b5", "\U0001f604")
x
#> [1] "one\ntwo" "one\ttwo" "µ"        "😄"
str_view(x)
#> [1] │ one
#>     │ two
#> [2] │ one{\t}two
#> [3] │ µ
#> [4] │ 😄

str_view() は、タブを識別しやすくするために中括弧（{ }）を使用することに注意してください 。テキストを扱う際の課題の一つは、さまざまな方法で空白がテキストに含まれてしまうことですが、この表示は何か奇妙なことが起きていると認識するのに役立ちます 。

14.2.4 演習問題

  1. 以下の値を含む文字列を作成してください。
    1. He said "That's amazing!"
    2. \a\b\c\d
    3. \\\\\\
  2. R セッションで文字列を作成し、印刷してください。特別な “\u00a0” はどうなりますか？ str_view() はそれをどのように表示しますか？ この特殊文字が何であるか、少しググって調べることができますか？
    x <- "This\u00a0is\u00a0tricky"

14.3 データから多くの文字列を作成する

「手動」で一つか二つの文字列を作成する基本を学んだので、次は他の文字列から文字列を作成する詳細について説明します 。これは、自分で書いたテキストとデータフレームの文字列を組み合わせたいという一般的な問題を解決するのに役立ちます 。たとえば、挨拶を作成するために “Hello” と name 変数を組み合わせるかもしれません 。str_c() と str_glue() を使用してこれを行う方法と、これらを mutate() で使用する方法を紹介します 。そうすると、自然に summarize() でどの stringr 関数を使用するのかという疑問が生じるため、このセクションの最後では文字列の要約関数である str_flatten() について議論します 。

14.3.1 str_c()

str_c() は任意の数のベクトルを引数として受け取り、文字ベクトルを返します。

str_c("x", "y")
#> [1] "xy"
str_c("x", "y", "z")
#> [1] "xyz"
str_c("Hello ", c("John", "Susan"))
#> [1] "Hello John"  "Hello Susan"

str_c() はベース R の paste0() と非常によく似ていますが、リサイクルや欠損値の伝播に関する通常の tidyverse の規則に従うことで、mutate() と一緒に使用するように設計されています 。

df <- tibble(name = c("Flora", "David", "Terra", NA))
df |> mutate(greeting = str_c("Hi ", name, "!"))
#> # A tibble: 4 × 2
#>   name  greeting 
#>   <chr> <chr>    
#> 1 Flora Hi Flora!
#> 2 David Hi David!
#> 3 Terra Hi Terra!
#> 4 <NA>  <NA>

欠損値を別の方法で表示したい場合は、coalesce() を使用して置換してください 。何をしたいかに応じて、str_c() の内側または外側で使用します 。

df |>
  mutate(
    greeting1 = str_c("Hi ", coalesce(name, "you"), "!"),
    greeting2 = coalesce(str_c("Hi ", name, "!"), "Hi!")
  )
#> # A tibble: 4 × 3
#>   name  greeting1 greeting2
#>   <chr> <chr>     <chr>    
#> 1 Flora Hi Flora! Hi Flora!
#> 2 David Hi David! Hi David!
#> 3 Terra Hi Terra! Hi Terra!
#> 4 <NA>  Hi you!   Hi!

14.3.2 str_glue()

str_c() を使って多くの固定文字列と変数の文字列を混ぜていると、" を入力しすぎて、コードの全体的な目標が見えにくくなることに気づくでしょう 。代替のアプローチは、glue パッケージ が str_glue() を通じて提供しています 。これには特別な機能を持つ単一の文字列を渡します。{} の中にあるものはすべて、引用符の外にあるかのように評価されます 。

df |> mutate(greeting = str_glue("Hi {name}!"))
#> # A tibble: 4 × 2
#>   name  greeting 
#>   <chr> <glue>   
#> 1 Flora Hi Flora!
#> 2 David Hi David!
#> 3 Terra Hi Terra!
#> 4 <NA>  Hi NA!

見ての通り、str_glue() は現在、欠損値を文字列 "NA" に変換してしまいます。残念ながら、これは str_c() と矛盾しています 。

また、文字列の中に通常の { または } を含める必要がある場合はどうなるか疑問に思うかもしれません 。何らかの方法でエスケープする必要があると推測したなら、正しい道を進んでいます 。コツは、glue が少し異なるエスケープ技術を使用していることです。\ のような特殊文字を前に付ける代わりに、特殊文字を二重にします 。

df |> mutate(greeting = str_glue("{{Hi {name}!}}"))
#> # A tibble: 4 × 2
#>   name  greeting   
#>   <chr> <glue>     
#> 1 Flora {Hi Flora!}
#> 2 David {Hi David!}
#> 3 Terra {Hi Terra!}
#> 4 <NA>  {Hi NA!}

14.3.3 str_flatten()

str_c() と str_glue() は、出力が入力と同じ長さであるため、mutate() とうまく機能します 。もし、常に単一の文字列を返すような、summarize() とうまく機能する関数が必要な場合はどうすればよいでしょうか？ それが str_flatten() の役割です 。これは文字ベクトルを受け取り、ベクトルの各要素を単一の文字列に結合します 。

str_flatten(c("x", "y", "z"))
#> [1] "xyz"
str_flatten(c("x", "y", "z"), ", ")
#> [1] "x, y, z"
str_flatten(c("x", "y", "z"), ", ", last = ", and ")
#> [1] "x, y, and z"

これにより、summarize() とうまく機能するようになります。

df <- tribble(
  ~ name, ~ fruit,
  "Carmen", "banana",
  "Carmen", "apple",
  "Marvin", "nectarine",
  "Terence", "cantaloupe",
  "Terence", "papaya",
  "Terence", "mandarin"
)
df |>
  group_by(name) |>
  summarize(fruits = str_flatten(fruit, ", "))
#> # A tibble: 3 × 2
#>   name    fruits                      
#>   <chr>   <chr>                       
#> 1 Carmen  banana, apple               
#> 2 Marvin  nectarine                   
#> 3 Terence cantaloupe, papaya, mandarin

14.3.4 演習問題

  1. 以下の入力に対して、paste0() と str_c() の結果を比較対照してください。
str_c("hi ", NA)
str_c(letters[1:2], letters[1:3])
  1. paste() と paste0() の違いは何ですか？ str_c() を使って paste() と同等のものをどのように再現できますか？
  2. 以下の式を str_c() から str_glue() に、あるいはその逆に変換してください。
    1. str_c("The price of ", food, " is ", price)
    2. str_glue("I'm {age} years old and live in {country}")
    3. str_c("\\section{", title, "}")

14.4 文字列からデータを抽出する

複数の変数が一つの文字列の中に詰め込まれていることは非常によくあります 。このセクションでは、それらを抽出するための4つの tidyr 関数の使用方法を学びます 。

  • df |> separate_longer_delim(col, delim)
  • df |> separate_longer_position(col, width)
  • df |> separate_wider_delim(col, delim, names)
  • df |> separate_wider_position(col, widths)

よく見ると、共通のパターンがあることがわかります。separate_ の後に longer または wider が続き、さらに _ の後に delim または position が続きます 。これは、これら4つの関数が2つの単純なプリミティブから構成されているためです 。

  • pivot_longer() や pivot_wider() と同様に、_longer 関数は新しい行を作成して入力データフレームを長くし、_wider 関数は新しい列を生成して入力データフレームを広くします 。
  • delim は ", " や " " のような区切り文字で文字列を分割します 。position は c(3, 5, 2) のような指定された幅で分割します 。

このファミリーの最後のメンバーである separate_wider_regex() については、Chapter 15 で戻ってきます 。これは wider 関数の中で最も柔軟ですが、使用する前に正規表現について知っておく必要があります 。続く2つのセクションでは、これらの separate 関数の基本的な考え方として、まず行への分割（こちらの方が少し単純です）、次に列への分割について説明します 。最後は、問題を診断するために wider 関数が提供するツールについて議論して締めくくります 。

14.4.1 行への分割

文字列を行に分割することは、構成要素の数が各行で異なる場合に最も有用です 。最も一般的なケースは、区切り文字に基づいて分割するために separate_longer_delim() を必要とする場合です 。

df1 <- tibble(x = c("a,b,c", "d,e", "f"))
df1 |>
  separate_longer_delim(x, delim = ",")
#> # A tibble: 6 × 1
#>   x    
#>   <chr>
#> 1 a    
#> 2 b    
#> 3 c    
#> 4 d    
#> 5 e    
#> 6 f

実際に separate_longer_position() を目にすることは稀ですが、各文字が一つの値を記録するために使用されている非常にコンパクトな形式を使用している古いデータセットもあります。

df2 <- tibble(x = c("1211", "131", "21"))
df2 |>
  separate_longer_position(x, width = 1)
#> # A tibble: 9 × 1
#>   x    
#>   <chr>
#> 1 1    
#> 2 2    
#> 3 1    
#> 4 1    
#> 5 1    
#> 6 3    
#> # ℹ 3 more rows

14.4.2 列への分割

文字列を列に分割することは、各文字列に固定数の構成要素があり、それらを列に広げたい場合に最も有用です 。これらは longer に対応するものよりも少し複雑です。なぜなら列に名前を付ける必要があるからです 。たとえば、以下のデータセットでは、x は "." で区切られたコード、版数、および年で構成されています 。separate_wider_delim() を使用するには、区切り文字と名前を二つの引数で渡します 。

df3 <- tibble(x = c("a10.1.2022", "b10.2.2011", "e15.1.2015"))
df3 |>
  separate_wider_delim(
    x,
    delim = ".",
    names = c("code", "edition", "year")
  )
#> # A tibble: 3 × 3
#>   code  edition year 
#>   <chr> <chr>   <chr>
#> 1 a10   1       2022 
#> 2 b10   2       2011 
#> 3 e15   1       2015

特定の断片が不要な場合は、NA という名前を使用して結果から除外できます 。

df3 |>
  separate_wider_delim(
    x,
    delim = ".",
    names = c("code", NA, "year")
  )
#> # A tibble: 3 × 2
#>   code  year 
#>   <chr> <chr>
#> 1 a10   2022 
#> 2 b10   2011 
#> 3 e15   2015

separate_wider_position() は、通常、各列の幅を指定したいため、少し動作が異なります 。名前付き整数ベクトルを渡します。ここで名前は新しい列の名前になり、値はその列が占める文字数です 。名付けないことで、出力から値を除外できます 。

df4 <- tibble(x = c("202215TX", "202122LA", "202325CA"))
df4 |>
  separate_wider_position(
    x,
    widths = c(year = 4, age = 2, state = 2)
  )
#> # A tibble: 3 × 3
#>   year  age   state
#>   <chr> <chr> <chr>
#> 1 2022  15    TX   
#> 2 2021  22    LA   
#> 3 2023  25    CA

14.4.3 幅を広げる際の問題の診断

separate_wider_delim() は、固定され既知の列セットを必要とします 。もし、一部の行が期待される断片の数を持っていない場合はどうなるでしょうか？ 断片が少なすぎるか多すぎるかという二つの問題が考えられるため、separate_wider_delim() は too_few と too_many という二つの引数を提供しています 。

まず、以下のサンプルデータセットを使用して too_few（少なすぎる）のケースを見てみましょう 。

df <- tibble(a = c("1-1-1", "1-1-2", "1-3", "1-3-2", "1"))

df |>
  separate_wider_delim(
    a,
    delim = "-",
    names = c("x", "y", "z")
  )
#> Error in `separate_wider_delim()`:
#> ! Expected 3 pieces in each element of `a`.
#> ! 2 values were too short.
#> ℹ Use `too_few = "debug"` to diagnose the problem.
#> ℹ Use `too_few = "align_start"/"align_end"` to silence this message.

エラーが発生しますが、そのエラーはどう進めればよいかについての提案をしてくれます 。まず、問題をデバッグすることから始めましょう。

debug <- df |>
  separate_wider_delim(
    a,
    delim = "-",
    names = c("x", "y", "z"),
    too_few = "debug"
  )
#> Warning: Debug mode activated: adding variables `a_ok`, `a_pieces`, and
#> `a_remainder`.
debug
#> # A tibble: 5 × 7
#>   x     y     z     a     a_ok  a_pieces a_remainder
#>   <chr> <chr> <chr> <chr> <lgl>    <int> <chr>      
#> 1 1     1     1     1-1-1 TRUE         3 ""         
#> 2 1     1     2     1-1-2 TRUE         3 ""         
#> 3 1     3     <NA>  1-3   FALSE        2 ""         
#> 4 1     3     2     1-3-2 TRUE         3 ""         
#> 5 1     <NA>  <NA>  1     FALSE        1 ""

デバッグモードを使用すると、出力に3つの追加列が追加されます。a_ok, a_pieces, a_remainder です（違う名前の変数を分割する場合は、異なる接頭辞がつきます）。ここで、a_ok を使うと失敗した入力をすぐに見つけることができます。

debug |> filter(!a_ok)
#> # A tibble: 2 × 7
#>   x     y     z     a     a_ok  a_pieces a_remainder
#>   <chr> <chr> <chr> <chr> <lgl>    <int> <chr>      
#> 1 1     3     <NA>  1-3   FALSE        2 ""         
#> 2 1     <NA>  <NA>  1     FALSE        1 ""

a_pieces は、期待された 3（names の長さ）に対して、いくつの断片が見つかったかを教えてくれます 。a_remainder は断片が少なすぎる場合には役に立ちませんが、すぐにまた登場します 。このデバッグ情報を見ることで、区切り文字の戦略に問題があることが判明したり、分割の前にもっと前処理が必要であることに気づいたりすることがあります 。その場合は、上流で問題を修正し、新しい問題がエラーとして検出されるように too_few = "debug" を削除してください 。別のケースでは、不足している部分を NA で埋めて先に進みたいこともあるでしょう 。それが too_few = "align_start" と too_few = "align_end" の役割で、NA をどこに置くかを制御できます 。

df |>
  separate_wider_delim(
    a,
    delim = "-",
    names = c("x", "y", "z"),
    too_few = "align_start"
  )
#> # A tibble: 5 × 3
#>   x     y     z    
#>   <chr> <chr> <chr>
#> 1 1     1     1    
#> 2 1     1     2    
#> 3 1     3     <NA> 
#> 4 1     3     2    
#> 5 1     <NA>  <NA>

断片が多すぎる（too many pieces）場合も同じ原則が適用されます 。

df <- tibble(a = c("1-1-1", "1-1-2", "1-3-5-6", "1-3-2", "1-3-5-7-9"))

df |>
  separate_wider_delim(
    a,
    delim = "-",
    names = c("x", "y", "z")
  )
#> Error in `separate_wider_delim()`:
#> ! Expected 3 pieces in each element of `a`.
#> ! 2 values were too long.
#> ℹ Use `too_many = "debug"` to diagnose the problem.
#> ℹ Use `too_many = "drop"/"merge"` to silence this message.

しかし今度は、結果をデバッグすると a_remainder の目的が見えてきます。

debug <- df |>
  separate_wider_delim(
    a,
    delim = "-",
    names = c("x", "y", "z"),
    too_many = "debug"
  )
#> Warning: Debug mode activated: adding variables `a_ok`, `a_pieces`, and
#> `a_remainder`.
debug |> filter(!a_ok)
#> # A tibble: 2 × 7
#>   x     y     z     a         a_ok  a_pieces a_remainder
#>   <chr> <chr> <chr> <chr>     <lgl>    <int> <chr>      
#> 1 1     3     5     1-3-5-6   FALSE        4 -6         
#> 2 1     3     5     1-3-5-7-9 FALSE        5 -7-9

多すぎる断片を処理するためのオプションは少し異なります。追加の断片を黙って「ドロップ（drop）」するか、それらすべてを最後の列に「マージ（merge）」するかのどちらかを選択できます 。

df |>
  separate_wider_delim(
    a,
    delim = "-",
    names = c("x", "y", "z"),
    too_many = "drop"
  )
#> # A tibble: 5 × 3
#>   x     y     z    
#>   <chr> <chr> <chr>
#> 1 1     1     1    
#> 2 1     1     2    
#> 3 1     3     5    
#> 4 1     3     2    
#> 5 1     3     5


df |>
  separate_wider_delim(
    a,
    delim = "-",
    names = c("x", "y", "z"),
    too_many = "merge"
  )
#> # A tibble: 5 × 3
#>   x     y     z    
#>   <chr> <chr> <chr>
#> 1 1     1     1    
#> 2 1     1     2    
#> 3 1     3     5-6  
#> 4 1     3     2    
#> 5 1     3     5-7-9

14.5 個々の文字

このセクションでは、文字列の中の個々の文字（letters）を扱うための関数を紹介します 。文字列の長さを求め、部分文字列を抽出し、プロットやテーブルで長い文字列を処理する方法を学びます 。

14.5.1 長さ

str_length() は、文字列の中の文字数を教えてくれます。

str_length(c("a", "R for data science", NA))
#> [1]  1 18 NA

これと count() を併用して米国の赤ちゃんの名（babynames）の長さの分布を調べ、filter() を使って最も長い名前を確認することができます。それらは15文字であることがわかります 。

babynames |>
  count(length = str_length(name), wt = n)
#> # A tibble: 14 × 2
#>   length        n
#>    <int>    <int>
#> 1      2   338150
#> 2      3  8589596
#> 3      4 48506739
#> 4      5 87011607
#> 5      6 90749404
#> 6      7 72120767
#> # ℹ 8 more rows

babynames |>
  filter(str_length(name) == 15) |>
  count(name, wt = n, sort = TRUE)
#> # A tibble: 34 × 2
#>   name                n
#>   <chr>           <int>
#> 1 Franciscojavier   123
#> 2 Christopherjohn   118
#> 3 Johnchristopher   118
#> 4 Christopherjame   108
#> 5 Christophermich    52
#> 6 Ryanchristopher    45
#> # ℹ 28 more rows

14.5.2 部分抽出

str_sub(string, start, end) を使用して文字列の一部を抽出できます。ここで start と end は、部分文字列を開始および終了する位置です 。start と end 引数はその値を含みます。したがって、返される文字列の長さは end - start + 1 になります 。

x <- c("Apple", "Banana", "Pear")
str_sub(x, 1, 3)
#> [1] "App" "Ban" "Pea"

負の値を使用して、文字列の末尾から数えることができます。-1 は最後の文字、-2 は最後から2番目の文字などです 。

str_sub(x, -3, -1)
#> [1] "ple" "ana" "ear"

文字列が短すぎても str_sub() は失敗しません。可能な限り多くの文字を返します 。

str_sub("a", 1, 5)
#> [1] "a"

str_sub() を mutate() と併用して、各名前の最初と最後の文字を見つけることができます 。

babynames |>
  mutate(
    first = str_sub(name, 1, 1),
    last = str_sub(name, -1, -1)
  )
#> # A tibble: 1,924,665 × 7
#>    year sex   name          n   prop first last 
#>   <dbl> <chr> <chr>     <int>  <dbl> <chr> <chr>
#> 1  1880 F     Mary       7065 0.0724 M     y    
#> 2  1880 F     Anna       2604 0.0267 A     a    
#> 3  1880 F     Emma       2003 0.0205 E     a    
#> 4  1880 F     Elizabeth  1939 0.0199 E     h    
#> 5  1880 F     Minnie     1746 0.0179 M     e    
#> 6  1880 F     Margaret   1578 0.0162 M     t    
#> # ℹ 1,924,659 more rows

  2. str_length() と str_sub() を使用して、各赤ちゃんの名から真ん中の文字を抽出してください 。文字列の文字数が偶数の場合はどうしますか？
  3. 時間の経過とともに、赤ちゃんの名（babynames）の長さに大きな傾向はありますか？ 最初と最後の文字の人気についてはどうですか？

14.6 英語以外のテキスト

ここまでは、二つの理由から特に扱いやすい英語のテキストに焦点を当ててきました 。第一に、英語のアルファベットは比較的単純です。26文字しかありません 。第二に（そしておそらくより重要なことに）、現在私たちが使用しているコンピュータ・インフラは、主に英語を話す人々によって設計されたからです 。残念ながら、英語以外の言語をすべて網羅する余裕はありません 。それでも、遭遇する可能性のある最大の課題として、エンコーディング、文字のバリエーション、およびロケール依存の関数に注意を向けておきたいと思います 。

14.6.1 エンコーディング

英語以外のテキストを扱う際、最初の課題はしばしばエンコーディング（encoding）です 。何が起きているのかを理解するために、コンピュータがどのように文字列を表現しているかを掘り下げる必要があります 。R では、charToRaw() を使用して文字列の基礎となる表現を取得できます 。

charToRaw("Hadley")
#> [1] 48 61 64 6c 65 79

これらの6つの16進数は、それぞれ一つの文字を表しています。48 は H、61 は a といった具合です 。16進数から文字へのマッピングはエンコーディングと呼ばれ、この場合、そのエンコーディングは ASCII と呼ばれます 。ASCII は、アメリカ情報交換標準符号（American Standard Code for Information Interchange）であるため、英語の文字を表現するには非常に適しています 。英語以外の言語では、それほど簡単ではありません 。コンピュータの初期には、英語以外の文字をエンコードするための競合する規格が多数ありました 。たとえば、ヨーロッパ向けには2種類の異なるエンコーディングがありました。Latin1（ISO-8859-1）は西欧諸語に使用され、Latin2（ISO-8859-2）は中欧諸語に使用されました 。Latin1 では、バイト b1 は “±” ですが、Latin2 では “ą” です！

幸いなことに、今日ではほぼどこでもサポートされている一つの標準があります。UTF-8 です 。UTF-8 は、今日人間が使用するほぼすべての文字と、絵文字のような多くの追加記号をエンコードできます 。readr はあらゆる場所で UTF-8 を使用します 。これは良いデフォルト設定ですが、UTF-8 を使用していない古いシステムで作成されたデータでは失敗します 。そうなると、文字列を印刷したときに奇妙に見えます 。一つか二つの文字が化けているだけのこともあれば、完全に文字化けしてしまうこともあります 。

たとえば、珍しいエンコーディングを持つ二つのインライン CSV を示します 。

x1 <- "text\nEl Ni\xf1o was particularly bad this year"
read_csv(x1)$text
#> [1] "El Ni\xf1o was particularly bad this year"

x2 <- "text\n\x82\xb1\x82\xf1\x82\xc9\x82\xbf\x82\xcd"
read_csv(x2)$text
#> [1] "\x82\xb1\x82\xf1\x82ɂ\xbf\x82\xcd"

これらを正しく読み取るには、locale 引数でエンコーディングを指定します 。

read_csv(x1, locale = locale(encoding = "Latin1"))$text
#> [1] "El Niño was particularly bad this year"

read_csv(x2, locale = locale(encoding = "Shift-JIS"))$text
#> [1] "こんにちは"

正しいエンコーディングをどうやって見つければよいでしょうか？ 運が良ければ、データのドキュメントのどこかに記載されています 。残念ながら、そうしたケースは稀なので、readr はそれを突き止めるのに役立つ guess_encoding() を提供しています 。これは万能ではありませんし、（今回のように短いものではなく）大量のテキストがある場合によりうまく機能しますが、出発点としては妥当です 。正しいものが見つかるまで、いくつかの異なるエンコーディングを試してみることを覚悟してください。

エンコーディングは豊かで複雑なトピックです。ここでは表面をなぞったに過ぎません 。さらに詳しく学びたい場合は、http://kunststube.net/encoding/ にある詳細な説明を読むことをお勧めします 。

14.6.2 文字のバリエーション

アクセント記号のある言語を扱う場合、文字の位置を決定する際に（例：str_length() や str_sub() で）大きな課題が生じます。なぜなら、アクセント付きの文字は、一つの独立した文字（例：ü）としてエンコードされることもあれば、アクセントのない文字（例：u）と発音区別符号（例：¨）を組み合わせた二つの文字としてエンコードされることもあるからです 。たとえば、このコードは、見た目が同一な二つの ü の表現方法を示しています。

u <- c("\u00fc", "u\u0308")
str_view(u)
#> [1] │ ü
#> [2] │ ü

しかし、両方の文字列は長さが異なり、最初の文字も異なります。

str_length(u)
#> [1] 1 2
str_sub(u, 1, 1)
#> [1] "ü" "u"

最後に、これらを == で比較すると「異なる」と解釈されますが、stringr の便利な str_equal() 関数は、両方が同じ外見であることを認識します。

u[[1]] == u[[2]]
#> [1] FALSE

str_equal(u[[1]], u[[2]])
#> [1] TRUE

14.6.3 ロケール依存の関数

最後に、動作がロケール（locale）に依存する stringr 関数がいくつかあります 。ロケールは言語に似ていますが、言語内の地域バリエーションを処理するためのオプションの地域指定子を含みます 。ロケールは、小文字の言語略称で指定され、オプションで _ と大文字の地域識別子が続きます 。たとえば、“en” は英語、“en_GB” はイギリス英語、“en_US” はアメリカ英語です 。自分の言語のコードを知らない場合は、Wikipedia に良いリストがありますし、stringi::stri_locale_list() で stringr がサポートしているものを確認できます 。

ベース R の文字列関数は、オペレーティングシステムによって設定されたロケールを自動的に使用します 。これは、ベース R 関数があなたの言語で期待通りの動作をすることを意味しますが、別の国に住んでいる誰かとコードを共有した場合、動作が異なる可能性があります 。この問題を避けるため、stringr はデフォルトで “en” ロケールを使用して英語の規則に従い、それを上書きするには locale 引数を指定することを要求します 。

幸いなことに、ロケールが本当に重要になるのは、大文字小文字の変換と並べ替え（ソート）の二つの関数セットだけです 。大文字小文字を変換する規則は言語によって異なります。たとえば、トルコ語には点の有無で区別される二つの i があります 。それらは二つの異なる文字であるため、大文字への変換も異なります。

str_to_upper(c("i", "ı"))
#> [1] "I" "I"
str_to_upper(c("i", "ı"), locale = "tr")
#> [1] "İ" "I"

文字列の並べ替えはアルファベットの順序に依存しますが、アルファベットの順序はすべての言語で同じではありません ！ たとえばチェコ語では、“ch” はアルファベットで h の後に現れる複合文字です 。

str_sort(c("a", "c", "ch", "h", "z"))
#> [1] "a"  "c"  "ch" "h"  "z"
str_sort(c("a", "c", "ch", "h", "z"), locale = "cs")
#> [1] "a"  "c"  "h"  "ch" "z"

これは dplyr::arrange() で文字列を並べ替える際にも関係するため、そこにも locale 引数があります 。
