6  ワークフロー：スクリプトとプロジェクト

この章では、コードを整理するための2つの不可欠なツール、スクリプトとプロジェクトを紹介します 。

6.1 スクリプト

これまでは、コードを実行するためにコンソールを使用してきました 。使い始めとしては素晴らしい場所ですが、より複雑な ggplot2 グラフィックスや長い dplyr パイプラインを作成するようになると、すぐに手狭に感じるようになるでしょう 。作業スペースを広げるには、スクリプトエディタを使用してください 。[File] メニューをクリックして [New File]、次に [R script] を選択するか、キーボードショートカット Cmd/Ctrl + Shift + N を使用して開きます。すると、Figure 6.1 のように4つのペインが表示されます 。スクレイプトエディタは、コードを試行錯誤するのに最適な場所です 。何かを変更したいとき、すべてを再入力する必要はありません。スクリプトを編集して再実行するだけです 。そして、期待通りに動作するコードが書けたら、後で簡単に戻ってこられるようにスクリプトファイルとして保存できます 。

6.1.1 コードの実行

スクリプトエディタは、複雑な ggplot2 プロットの構築や、長い一連の dplyr 操作を行うのに非常に適した場所です 。スクリプトエディタを効果的に使用するための鍵は、最も重要なキーボードショートカットの一つである Cmd/Ctrl + Enter を覚えることです 。これは、コンソールで現在の R 式を実行します。例えば、以下のコードを見てみましょう。

library(dplyr)
library(nycflights13)

not_cancelled <- flights |>
  filter(!is.na(dep_delay)█, !is.na(arr_delay))

not_cancelled |>
  group_by(year, month, day) |>
  summarize(mean = mean(dep_delay))

カーソルが █ の位置にある場合、Cmd/Ctrl + Enter を押すと not_cancelled を生成するコマンド全体が実行されます 。また、カーソルは次のステートメント（not_cancelled |> で始まる行）に移動します 。これにより、Cmd/Ctrl + Enter を繰り返し押すだけで、スクリプト全体を簡単に進めることができます 。コードを式ごとに実行する代わりに、Cmd/Ctrl + Shift + S を使ってスクリプト全体を一段階で実行することもできます。これを定期的に行うことは、コードの重要な部分がすべてスクリプトに記述されているかを確認するための優れた方法です 。スクリプトは常に、必要なパッケージの読み込みから始めることをお勧めします 。そうすれば、コードを他の人と共有したときに、その人がどのパッケージをインストールする必要があるか一目でわかります 。ただし、共有するスクリプトに install.packages() を含めてはいけません 。注意を払っていない人のコンピュータで設定を変更してしまうようなスクリプトを渡すのは、配慮に欠けます ！今後の章を進める際は、スクリプトエディタから始めて、キーボードショートカットを練習することを強くお勧めします 。時間が経てば、このようにコードをコンソールに送ることは、意識しなくても自然に行えるようになるでしょう 。

6.1.2 RStudio の診断機能

スクリプトエディタでは、RStudio が構文エラーを赤い波線とサイドバーのバツ印でハイライトします。

バツ印の上にマウスを置くと、何が問題かが表示されます。

RStudio は潜在的な問題についても知らせてくれます。

6.1.3 保存と命名

RStudio は終了時にスクリプトエディタの内容を自動的に保存し、再起動時に自動的にリロードします 。それにもかかわらず、「Untitled1」「Untitled2」「Untitled3」といった名前は避け、スクリプトを保存して内容がわかる名前を付けるのが良い習慣です 。ファイルを code.R や myscript.R と名付けたくなるかもしれませんが、ファイル名を選ぶ前にもう少し慎重に考えるべきです 。ファイル命名における3つの重要な原則は以下の通りです。

  1. ファイル名はマシンリーダブル（機械可読）であるべきです。スペース、記号、特殊文字を避けてください 。大文字・小文字の違いだけでファイルを区別しようとしないでください。
  2. ファイル名はヒューマンリーダブル（人間可読）であるべきです。ファイル名を使って、そのファイルの中に何が入っているかを説明してください 。
  3. ファイル名はデフォルトの並び順（並び替え）に適したものであるべきです。ファイル名を数字で始めることで、アルファベット順に並べたときに、使用される順番通りに並ぶようにします 。

例えば、プロジェクトフォルダに以下のようなファイルがあるとします 。

alternative model.R
code for exploratory analysis.r
finalreport.qmd
FinalReport.qmd
fig 1.png
Figure_02.png
model_first_try.R
run-first.r
temp.txt

ここにはさまざまな問題があります。どのファイルを最初に実行すべきか判断しにくく、ファイル名にスペースが含まれており、大文字・小文字が異なるだけの同じ名前のファイルが2つあり（finalreport と FinalReport 1）、いくつかの名前はその内容を説明していません（run-first や temp）。

【Geminiの補足】: ファイル名に「final（最終）」という言葉を入れると、その後「final_v2.R」「final_really_final.R」のように、皮肉にも決して「最終」にならないファイルが増殖していくという、データ分析やレポート作成における有名な「あるある」ネタです。

同じファイル群を整理・命名するより良い方法を以下に示します。

01-load-data.R
02-exploratory-analysis.R
03-model-approach-1.R
04-model-approach-2.R
fig-01.png
fig-02.png
report-2022-03-20.qmd
report-2022-04-02.qmd
report-draft-notes.txt

主要なスクリプトに番号を振ることで、どの順番で実行すべきかが明白になり、一貫した命名規則によって何が異なるのかが見やすくなります 。さらに、図には同様のラベルが付けられ、レポートはファイル名に含まれる日付で区別され、temp は内容をよりよく表す report-draft-notes に変更されています 。ディレクトリ内に大量のファイルがある場合は、さらに一歩進んで、種類（スクリプト、図など）ごとに異なるディレクトリに配置することをお勧めします 。

6.2 プロジェクト

いつか、R を終了して他のことを行い、後で分析に戻る必要がある日が来ます 。いつか、複数の分析を並行して行い、それらを別々に保ちたいと思う日が来ます 。いつか、外部からデータを R に取り込み、数値結果や図を R から外部へ送り出す必要がある日が来ます 。これらの現実的な状況に対処するには、2つの決定を下す必要があります。

  1. 何が「真実のソース (Source of truth)」か？ 何を、起きたことの永続的な記録として保存しますか ？
  2. 分析はどこに存在しますか ？

6.2.1 何が「真実のソース」か？

初心者の方は、分析全体で作成したすべてのオブジェクトを含む現在の環境（Environment）に頼っても構いません 。しかし、より大きなプロジェクトに取り組んだり、他の人と協力したりしやすくするためには、R スクリプトを「真実のソース」とすべきです 。R スクリプト（とデータファイル）があれば、環境を再現できます 。環境だけでは R スクリプトを再現するのははるかに困難です。記憶を頼りに大量のコードを再入力するか（その過程で必ず間違いを犯します）、R の履歴（History）を注意深く掘り起こす必要があります 。R スクリプトを分析の真実のソースとして保つために、セッション間でワークスペースを保存しないように RStudio に指示することを強くお勧めします 。これは、usethis::use_blank_slate() 2 を実行するか、Figure 6.2 に示すオプションを真似ることで設定できます 。これにより、RStudio を再起動したときに前回実行したコードが記憶されず、作成したオブジェクトや読み込んだデータセットも利用できなくなるため、短期的には苦痛を感じるでしょう 。しかし、この短期的な苦痛は、コードの中にすべての重要な手順を記録することを強制するため、長期的な苦悩を救ってくれます 。重要な計算の結果だけを環境に保存していて、その計算式自体をコードに保存していなかったことに3ヶ月後に気づくほど最悪なことはありません 。

エディタにコードの重要な部分が保存されているかを確認するために役立つ、強力なキーボードショートカットのペアがあります。

  1. Cmd/Ctrl + Shift + 0/F10 を押して R を再起動します。
  2. Cmd/Ctrl + Shift + S を押して現在のスクリプトを再実行します 。

私たちは共同でこのパターンを週に何百回も使用しています 。あるいは、キーボードショートカットを使わない場合は、[Session] > [Restart R] に移動し、現在のスクリプトをハイライトして再実行することもできます 。

NoteRStudio server

RStudio server を使用している場合、R セッションはデフォルトで再起動されることはありません 。RStudio server のタブを閉じると R を終了したように感じるかもしれませんが、サーバーは実際にはバックグラウンドで実行し続けています 。次に戻ったときは、離れたときと全く同じ場所にいます 。そのため、常に白紙の状態から始められるよう、定期的に R を再起動することがさらに重要になります 。

6.2.2 分析はどこに存在しますか？

R には、作業ディレクトリ（working directory）という強力な概念があります 。これは R が読み込みを指示されたファイルを探す場所であり、保存を指示されたファイルを置く場所です 。RStudio はコンソールの最上部に現在の作業ディレクトリを表示します。

また、getwd() を実行することで R コードとしてこれを出力できます。

getwd()
#> [1] "/Users/hadley/Documents/r4ds"

この R セッションでは、現在の作業ディレクトリ（「ホーム」と考えてください）は hadley の Documents フォルダ内の r4ds というサブフォルダにあります 。あなたが実行したとき、このコードは異なる結果を返します。なぜなら、あなたのコンピュータは Hadley のものとは異なるディレクトリ構造を持っているからです ！ R の初心者としては、作業ディレクトリをホームディレクトリやドキュメントディレクトリ、あるいはコンピュータ上の適当なディレクトリにしておいても構いません 。しかし、あなたは本書を数章分読み進めており、もう初心者ではありません 。今後は、プロジェクトをディレクトリごとに整理し、プロジェクトに取り組む際には R の作業ディレクトリを関連するディレクトリに設定するように進化すべきです 。R の内部から作業ディレクトリを設定することもできますが、お勧めしません。

setwd("/path/to/my/CoolProject")

もっと良い方法があります。それは、エキスパートのように R の仕事を管理するための道でもあります 。その方法が RStudio プロジェクト (RStudio project) です。

6.2.3 RStudio プロジェクト

特定のプロジェクトに関連するすべてのファイル（入力データ、R スクリプト、分析結果、図）を一箇所にまとめておくことは、非常に賢明かつ一般的な習慣であるため、RStudio にはプロジェクトを通じた組み込みのサポートがあります 。本書の残りの部分を進める際に使用するためのプロジェクトを作成してみましょう > [New Project] をクリックし、Figure 6.3 に示す手順に従ってください。

プロジェクト名を r4ds とし、プロジェクトをどのサブディレクトリに置くか慎重に考えてください。分別のある場所に保存しないと、将来見つけるのが難しくなります ！このプロセスが完了すると、本書専用の新しい RStudio プロジェクトが作成されます 。プロジェクトの「ホーム」が現在の作業ディレクトリであることを確認してください。

getwd()
#> [1] /Users/hadley/Documents/r4ds

次に、スクレイプトエディタに以下のコマンドを入力し、“diamonds.R” という名前でファイルを保存してください 。その後、“data” という名前の新しいフォルダを作成します。これは、RStudio の [Files] ペインにある [New Folder] ボタンをクリックすることで行えます 。最後に、スクリプト全体を実行します。これにより、プロジェクトディレクトリに PNG ファイルと CSV ファイルが保存されます 。詳細は後で学べるので、今は気にしないでください。

library(tidyverse)

ggplot(diamonds, aes(x = carat, y = price)) +
  geom_hex()
ggsave("diamonds.png")

write_csv(diamonds, "data/diamonds.csv")

RStudio を終了します 。プロジェクトに関連付けられたフォルダを調べてください。.Rproj ファイルがあることに気づくはずです。そのファイルをダブルクリックしてプロジェクトを再度開きます 。離れたときと同じ状態に戻っていることに注目してください。作業ディレクトリやコマンド履歴は同じで、作業中だったファイルもすべて開いたままです 。ただし、上記の指示に従ったため、環境は完全に新しい状態になっており、白紙の状態から始めることが保証されています 。OS ごとの方法でコンピュータ内の diamonds.png を検索すると、PNG ファイル（当然ですが）だけでなく、それを作成したスクリプト（diamonds.R）も見つかるはずです 。これは大きな勝利です ！いつか、図を再作成したり、それがどこから来たのかを理解したくなる日が来ます 。マウスやクリップボードではなく、R コードを使用して図をファイルに保存することを厳守すれば、過去の仕事を簡単に再現できるようになります ！

6.2.4 相対パスと絶対パス

プロジェクトの中にいるときは、絶対パスではなく、常に相対パスだけを使うべきです 。違いは何でしょうか？ 相対パスは作業ディレクトリ、つまりプロジェクトのホームからの相対的な位置を示します 。Hadley が上で data/diamonds.csv と書いたとき、それは /Users/hadley/Documents/r4ds/data/diamonds.csv のショートカットでした 。しかし重要なことに、Mine がこのコードを自分のコンピュータで実行した場合、それは /Users/Mine/Documents/r4ds/data/diamonds.csv を指すことになります 。これが相対パスが重要である理由です。R プロジェクトフォルダがどこに移動しても機能します 。

絶対パスは、作業ディレクトリに関係なく常に同じ場所を指します 。オペレーティングシステムによって見た目が少し異なります 。Windows ではドライブ文字（例：C:）または2つのバックスラッシュ（例：\\servername）で始まり、Mac/Linux ではスラッシュ “/”（例：/users/hadley）で始まります 。スクリプトの中で絶対パスを決して使ってはいけません。共有が困難になるからです。あなたと全く同じディレクトリ構成を持っている人は他にいません 。

OS 間のもう一つの重要な違いは、パスの構成要素をどのように区切るかです 。Mac と Linux はスラッシュ（例：data/diamonds.csv）を使い、Windows はバックスラッシュ（例：data\diamonds.csv）を使います 。R はいずれの形式でも動作しますが（あなたが現在どのプラットフォームを使っていようとも）、残念ながらバックスラッシュは R にとって特別な意味を持つため、パスに1つのバックスラッシュを入れるには、2つ入力する必要があります ！それは生活を苛立たせるものにするので、私たちは常に Linux/Mac スタイルのフォワードスラッシュ（/）を使うことをお勧めします 。

6.3 演習問題

  1. RStudio Tips の Twitter アカウント（現在は X） https://twitter.com/rstudiotips に行き、興味深いヒントを一つ見つけてください 。実際に使ってみましょう！
  2. RStudio の診断機能は、他にどのようなよくある間違いを報告してくれますか？ https://support.posit.co/hc/en-us/articles/205753617-Code-Diagnostics を読んで調べてください 。

6.4 まとめ

この章では、R コードをスクリプト（ファイル）とプロジェクト（ディレクトリ）で整理する方法を学びました 。コードスタイルと同様に、最初は面倒な作業に感じるかもしれません 。しかし、複数のプロジェクトにわたってより多くのコードが蓄積されるにつれて、事前のわずかな整理が将来の時間を大幅に節約してくれることに感謝するようになるでしょう 。まとめると、スクリプトとプロジェクトは、将来役立つ強固なワークフローを提供します。

  • データ分析プロジェクトごとに1つの RStudio プロジェクトを作成してください 。
  • プロジェクト内にスクリプトを（わかりやすい名前で）保存し、編集し、部分的に、あるいは全体を実行してください 。R を頻繁に再起動して、すべてがスクリプトに記録されていることを確認してください 。
  • 絶対パスではなく、相対パスのみを使用してください 。

そうすれば、必要なものはすべて一箇所にあり、取り組んでいる他のすべてのプロジェクトから綺麗に分離されます 。これまでは、R パッケージの中にまとめられたデータセットを使ってきました 。これによって準備されたデータで練習しやすくなりましたが、当然ながらあなたのデータはそのようには利用できません 。そこで次の章では、readr パッケージを使ってディスクから R セッションにデータを読み込む方法を学びます。

  1. 名前に「final（最終）」を使うことで運命を試そうとしている（結局終わらない）ことは言うまでもありません 😆 コミックの Piled Higher and Deeper には、これに関する楽しいエピソードがあります 。↩︎

  2. もし usethis がインストールされていない場合は、install.packages("usethis") でインストールできます。↩︎
