23  階層データ

23.1 はじめに

この章では、データレクタングリング（rectangling：矩形化）の技術を学びます。これは、本質的に階層的、あるいはツリー状であるデータを取り込み、行と列で構成される長方形（矩形）のデータフレームに変換する作業のことです。これは非常に重要です。なぜなら、特にウェブから取得するデータを扱う場合、階層データは驚くほど一般的だからです。

レクタングリングについて学ぶには、まず階層データを可能にするデータ構造である「リスト」について知る必要があります。次に、tidyrの2つの重要な関数、tidyr::unnest_longer() と tidyr::unnest_wider() について学びます。その後、いくつかのケーススタディを紹介し、これらのシンプルな関数を繰り返し適用して現実の問題を解決する方法を示します。最後に、階層型データセットの最も頻繁なソースであり、ウェブ上でのデータ交換の一般的な形式であるJSONについてお話しして締めくくります。

23.1.1 事前準備

この章では、tidyverseのコアメンバーであるtidyrパッケージの多くの関数を使用します。また、レクタングリングの練習用に興味深いデータセットを提供してくれるrepurrrsiveパッケージを使用し、最後にjsonliteを使用してJSONファイルをRのリストに読み込みます。

library(tidyverse)
library(repurrrsive)
#> Warning: package 'repurrrsive' was built under R version 4.5.3
library(jsonlite)

23.2 リスト

これまでは、整数、数値、文字、日時、因子などの単純なベクトルを含むデータフレームを扱ってきました。これらのベクトルは、すべての要素が同じデータ型であるため、「均一（homogeneous）」で単純です。同じベクトル内に異なる型の要素を格納したい場合は、list() で作成するリストが必要になります。

x1 <- list(1:4, "a", TRUE)
x1
#> [[1]]
#> [1] 1 2 3 4
#> 
#> [[2]]
#> [1] "a"
#> 
#> [[3]]
#> [1] TRUE

リストの構成要素、つまり子（children）に名前を付けると便利なことがよくあります。これは、tibbleの列に名前を付けるのと同じ方法で行えます。

x2 <- list(a = 1:2, b = 1:3, c = 1:4)
x2
#> $a
#> [1] 1 2
#> 
#> $b
#> [1] 1 2 3
#> 
#> $c
#> [1] 1 2 3 4

これらの非常に単純なリストであっても、表示にはかなりのスペースを占有します。便利な代替手段は str() です。これは構造（structure）をコンパクトに表示し、内容よりも構造を強調します。

str(x1)
#> List of 3
#>  $ : int [1:4] 1 2 3 4
#>  $ : chr "a"
#>  $ : logi TRUE
str(x2)
#> List of 3
#>  $ a: int [1:2] 1 2
#>  $ b: int [1:3] 1 2 3
#>  $ c: int [1:4] 1 2 3 4

見ての通り、str() はリストの各子をそれぞれの行に表示します。名前がある場合は名前を表示し、次に型の略称、そして最初の数個の値を表示します。

23.2.1 階層

リストには、他のリストを含む任意の型のオブジェクトを含めることができます。これにより、階層構造（ツリー状の構造）を表現するのに適しています。

x3 <- list(list(1, 2), list(3, 4))
str(x3)
#> List of 2
#>  $ :List of 2
#>   ..$ : num 1
#>   ..$ : num 2
#>  $ :List of 2
#>   ..$ : num 3
#>   ..$ : num 4

これは、フラットなベクトルを生成する c() とは明らかに異なります。

c(c(1, 2), c(3, 4))
#> [1] 1 2 3 4

x4 <- c(list(1, 2), list(3, 4))
str(x4)
#> List of 4
#>  $ : num 1
#>  $ : num 2
#>  $ : num 3
#>  $ : num 4

リストが複雑になるにつれて、階層を一目で確認できる str() の有用性が増します。

x5 <- list(1, list(2, list(3, list(4, list(5)))))
str(x5)
#> List of 2
#>  $ : num 1
#>  $ :List of 2
#>   ..$ : num 2
#>   ..$ :List of 2
#>   .. ..$ : num 3
#>   .. ..$ :List of 2
#>   .. .. ..$ : num 4
#>   .. .. ..$ :List of 1
#>   .. .. .. ..$ : num 5

リストがさらに大きく複雑になると、最終的に str() では対応しきれなくなり、View() に切り替える必要があります1。Figure 23.1 は、View(x5) を呼び出した結果を示しています。ビューアは最初はリストの最上位レベルのみを表示しますが、Figure 23.2 のようにコンポーネントをインタラクティブに展開して詳細を確認できます。また、RStudioは Figure 23.3 のように、その要素にアクセスするために必要なコードも表示してくれます。このコードの仕組みについては、Section 27.3 で改めて説明します。

23.2.2 リスト列

リストはtibbleの中に入れることもでき、その場合はリスト列（list-column）と呼びます。リスト列は、通常はtibbleに入れることができないようなオブジェクトを配置できるため便利です。特に、tidymodels のエコシステムでは、モデルの出力やリサンプリングの結果などをデータフレームに保存できるため、リスト列が多用されています。

リスト列の簡単な例を次に示します。

df <- tibble(
  x = 1:2,
  y = c("a", "b"),
  z = list(list(1, 2), list(3, 4, 5))
)
df
#> # A tibble: 2 × 3
#>       x y     z         
#>   <int> <chr> <list>    
#> 1     1 a     <list [2]>
#> 2     2 b     <list [3]>

tibble内のリストに特別なことは何もありません。他の列と同じように動作します。

df |>
  filter(x == 1)
#> # A tibble: 1 × 3
#>       x y     z         
#>   <int> <chr> <list>    
#> 1     1 a     <list [2]>

リスト列を使った計算は難しくなりますが、それは一般的にリストを使った計算が難しいためです。これについては Chapter 26 で改めて説明します。この章では、リスト列を通常の変数に「展開（unnesting）」することに焦点を当て、既存のツールを使えるようにします。

デフォルトの表示（print）メソッドは、内容の大まかな要約を表示するだけです。リスト列は任意に複雑になる可能性があるため、それを表示する適切な方法がないからです。内容を確認したい場合は、リスト列を1つだけ取り出して、前述の df |> pull(z) |> str() や df |> pull(z) |> View() などの手法を適用する必要があります。

NoteBase R

data.frame の列にリストを入れることは可能ですが、data.frame() はリストを「列のリスト」として扱うため、少し手間がかかります。

data.frame(x = list(1:3, 3:5))
#>   x.1.3 x.3.5
#> 1     1     3
#> 2     2     4
#> 3     3     5

リストを I() でラップすることで、data.frame() にリストを「行のリスト」として扱うよう強制できますが、結果の表示はあまり良くありません。

data.frame(
  x = I(list(1:2, 3:5)),
  y = c("1, 2", "3, 4, 5")
)
#>         x       y
#> 1    1, 2    1, 2
#> 2 3, 4, 5 3, 4, 5

tibbleでリスト列を使用する方が簡単です。なぜなら、tibble() はリストをベクトルと同じように扱い、表示メソッドもリストを考慮して設計されているからです。

23.3 展開（Unnesting）

リストとリスト列の基本を学んだところで、それらを通常の行や列に戻す方法を調べてみましょう。ここでは基本的な考え方を理解するために非常に単純なサンプルデータを使用し、次のセクションで実際のデータに切り替えます。

リスト列は主に「名前付き」と「名前なし」の2つの基本形式で提供されます。子が名前付きの場合、すべての行で同じ名前を持つ傾向があります。例えば df1 では、リスト列 y のすべての要素に a と b という名前の2つの要素があります。名前付きのリスト列は、自然に列へと展開されます。各名前付き要素が新しい名前付き列になります。

df1 <- tribble(
  ~x, ~y,
  1, list(a = 11, b = 12),
  2, list(a = 21, b = 22),
  3, list(a = 31, b = 32),
)

子が名前なしの場合、要素の数は行ごとに異なる傾向があります。例えば df2 では、リスト列 y の要素は名前がなく、長さは1から3まで変化します。名前なしのリスト列は、自然に行へと展開されます。子ごとに1つの行が作成されます。

df2 <- tribble(
  ~x, ~y,
  1, list(11, 12, 13),
  2, list(21),
  3, list(31, 32),
)

tidyrはこれら2つのケースに対して、unnest_wider() と unnest_longer() という2つの関数を提供しています。以下のセクションで、それらがどのように機能するかを説明します。

23.3.1 unnest_wider()

df1 のように、各行が同じ名前を持つ同じ数の要素を持っている場合、unnest_wider() を使用して各コンポーネントを独自の列に配置するのが自然です。

df1 |>
  unnest_wider(y)
#> # A tibble: 3 × 3
#>       x     a     b
#>   <dbl> <dbl> <dbl>
#> 1     1    11    12
#> 2     2    21    22
#> 3     3    31    32

デフォルトでは、新しい列の名前はリスト要素の名前からのみ取得されますが、names_sep 引数を使用して、列名と要素名を組み合わせるように指定することもできます。これは、重複する名前を区別するのに役立ちます。

df1 |>
  unnest_wider(y, names_sep = "_")
#> # A tibble: 3 × 3
#>       x   y_a   y_b
#>   <dbl> <dbl> <dbl>
#> 1     1    11    12
#> 2     2    21    22
#> 3     3    31    32

23.3.2 unnest_longer()

各行に名前のないリストが含まれている場合、unnest_longer() を使用して各要素を独自の行に配置するのが最も自然です。

df2 |>
  unnest_longer(y)
#> # A tibble: 6 × 2
#>       x     y
#>   <dbl> <dbl>
#> 1     1    11
#> 2     1    12
#> 3     1    13
#> 4     2    21
#> 5     3    31
#> 6     3    32

y の中の各要素に対して x が複製され、リスト列内の各要素につき1行の出力が得られることに注目してください。しかし、次の例のように要素の1つが空の場合はどうなるでしょうか？

df6 <- tribble(
  ~x, ~y,
  "a", list(1, 2),
  "b", list(3),
  "c", list()
)
df6 |> unnest_longer(y)
#> # A tibble: 3 × 2
#>   x         y
#>   <chr> <dbl>
#> 1 a         1
#> 2 a         2
#> 3 b         3

出力は0行になるため、その行は事実上消失します。その行を保持し、y に NA を入れたい場合は、keep_empty = TRUE を設定します。

23.3.3 型の不一致

異なる型のベクトルを含むリスト列を展開するとどうなるでしょうか？ 例えば、リスト列 y に2つの数値、1つの文字、1つの論理値が含まれている以下のデータセットを考えてみましょう。これらは通常、単一の列に混在させることはできません。

df4 <- tribble(
  ~x, ~y,
  "a", list(1),
  "b", list("a", TRUE, 5)
)

unnest_longer() は、行数を変更しながらも、列のセットは常に変更せずに維持します。では何が起こるのでしょうか？ unnest_longer() は、y のすべてを保持したままどのようにして5つの行を生成するのでしょうか？

df4 |>
  unnest_longer(y)
#> # A tibble: 4 × 2
#>   x     y        
#>   <chr> <list>   
#> 1 a     <dbl [1]>
#> 2 b     <chr [1]>
#> 3 b     <lgl [1]>
#> 4 b     <dbl [1]>

見ての通り、出力にはリスト列が含まれていますが、リスト列の各要素には単一の要素が含まれています。unnest_longer() はベクトルの共通の型を見つけることができないため、元の型をリスト列の中に保持します。

これは「列のすべての要素は同じ型でなければならない」という戒律を破っているのではないかと思うかもしれません。そんなことはありません。中身が異なる型であっても、すべての要素は「リスト」です。不一致な型の処理は難しく、詳細は問題の性質や目的によりますが、おそらく Chapter 26 のツールが必要になるでしょう。

23.3.4 その他の関数

tidyrには、この本では詳しく扱いませんが、レクタングリングに役立つ関数が他にもいくつかあります。

  • unnest_auto()：リスト列の構造に基づいて、unnest_longer() と unnest_wider() のどちらかを自動的に選択します。迅速な探索には最適ですが、最終的にはデータの構造を理解することを妨げ、コードの理解を難しくするため、あまりお勧めしません。
  • unnest()：行と列の両方を展開します。リスト列にデータフレームのような2次元構造が含まれている場合に便利です。この本では登場しませんが、tidymodels のエコシステムを使用すると遭遇するかもしれません。

これらの関数は、他人のコードを読んだり、珍しいレクタングリングの課題に取り組んだりする際に遭遇する可能性があるため、知っておいて損はありません。

23.3.5 練習問題

  1. df2 のような名前なしリスト列に unnest_wider() を使用するとどうなりますか？ どの引数が必須になりますか？ 欠損値はどうなりますか？
  2. df1 のような名前付きリスト列に unnest_longer() を使用するとどうなりますか？ 出力にはどのような追加情報が含まれますか？ その余分な詳細を抑制するにはどうすればよいですか？
  3. 時折、値が揃った複数のリスト列を持つデータフレームに遭遇することがあります。例えば以下のデータフレームでは、y と z の値が揃っています（つまり、行内では y と z は常に同じ長さであり、y の最初の値は z の最初の値に対応します）。このデータフレームに対して unnest_longer() を2回呼び出すとどうなりますか？ x と y の関係を維持するにはどうすればよいですか？（ヒント：ドキュメントを注意深く読んでください）。
    df4 <- tribble(
      ~x, ~y, ~z,
      "a", list("y-a-1", "y-a-2"), list("z-a-1", "z-a-2"),
      "b", list("y-b-1", "y-b-2", "y-b-3"), list("z-b-1", "z-b-2", "z-b-3")
    )

23.4 ケーススタディ

上記で使用した単純な例と実際のデータの主な違いは、実際のデータには通常、複数レベルの入れ子（ネスト）が含まれており、unnest_longer() や unnest_wider() を複数回呼び出す必要がある点です。それを実際に示すために、このセクションでは repurrrsive パッケージのデータセットを使用して、3つの現実的なレクタングリングの課題に取り組んでいきます。

23.4.1 非常に横長なデータ

まずは gh_repos から始めましょう。これは、GitHub APIを使用して取得された、GitHubリポジトリのコレクションに関するデータを含むリストです。非常に深くネストされたリストであるため、この本の中でその構造を示すのは困難です。先に進む前に、ご自身で View(gh_repos) を使って少し探索してみることをお勧めします。

gh_repos はリストですが、私たちのツールはリスト列に対して機能するため、まずはこれをtibbleに入れることから始めます。この列の名前を json と呼びますが、その理由は後ほど説明します。

repos <- tibble(json = gh_repos)
repos
#> # A tibble: 6 × 1
#>   json       
#>   <list>     
#> 1 <list [30]>
#> 2 <list [30]>
#> 3 <list [30]>
#> 4 <list [26]>
#> 5 <list [30]>
#> 6 <list [30]>

このtibbleには6つの行が含まれており、gh_repos の各子に対して1行ずつ対応しています。各行には、26行または30行の名前なしリストが含まれています。これらは名前なしなので、まずは unnest_longer() を使用して、各子を独自の行に配置します。

repos |>
  unnest_longer(json)
#> # A tibble: 176 × 1
#>   json             
#>   <list>           
#> 1 <named list [68]>
#> 2 <named list [68]>
#> 3 <named list [68]>
#> 4 <named list [68]>
#> 5 <named list [68]>
#> 6 <named list [68]>
#> # ℹ 170 more rows

一見すると、状況が改善されていないように見えるかもしれません。行数は増えましたが（6行から176行へ）、json の各要素は依然としてリストのままです。しかし、重要な違いがあります。今や各要素は名前付きリストになっているため、unnest_wider() を使用して各要素を独自の列に配置できるのです。

repos |>
  unnest_longer(json) |>
  unnest_wider(json)
#> # A tibble: 176 × 68
#>         id name        full_name         owner        private html_url       
#>      <int> <chr>       <chr>             <list>       <lgl>   <chr>          
#> 1 61160198 after       gaborcsardi/after <named list> FALSE   https://github…
#> 2 40500181 argufy      gaborcsardi/argu… <named list> FALSE   https://github…
#> 3 36442442 ask         gaborcsardi/ask   <named list> FALSE   https://github…
#> 4 34924886 baseimports gaborcsardi/base… <named list> FALSE   https://github…
#> 5 61620661 citest      gaborcsardi/cite… <named list> FALSE   https://github…
#> 6 33907457 clisymbols  gaborcsardi/clis… <named list> FALSE   https://github…
#> # ℹ 170 more rows
#> # ℹ 62 more variables: description <chr>, fork <lgl>, url <chr>, …

これでうまくいきましたが、結果は少し圧倒されるものです。列数が多すぎて、tibbleはすべての列を表示することさえできません！ names() を使えばすべて確認できます。ここでは最初の10個を見てみましょう。

repos |>
  unnest_longer(json) |>
  unnest_wider(json) |>
  names() |>
  head(10)
#>  [1] "id"          "name"        "full_name"   "owner"       "private"    
#>  [6] "html_url"    "description" "fork"        "url"         "forks_url"

面白そうなものをいくつか取り出してみましょう。

repos |>
  unnest_longer(json) |>
  unnest_wider(json) |>
  select(id, full_name, owner, description)
#> # A tibble: 176 × 4
#>         id full_name               owner             description             
#>      <int> <chr>                   <list>            <chr>                   
#> 1 61160198 gaborcsardi/after       <named list [17]> Run Code in the Backgro…

23.5 JSON

前のセクションのケーススタディはすべて、野生で採取されたJSONから提供されたものでした。JSONは JavaScript Object Notation の略で、ほとんどのウェブAPIがデータを返す方法です。JSONとRのデータ型は非常に似ていますが、完璧な1対1の対応関係があるわけではないため、JSONを理解しておくことは重要です。何か問題が発生したときに役立ちます。

23.5.1 データ型

JSONは、人間ではなく機械が簡単に読み書きできるように設計されたシンプルな形式です。6つの主要なデータ型があります。そのうち4つはスカラーです。

  • 最も単純な型はヌル（null）で、Rの NA と同じ役割を果たします。データの欠如を表します。
  • 文字列（string）はRの文字列によく似ていますが、常に二重引用符を使用する必要があります。
  • 数値（number）はRの数値に似ており、整数表記（例：123）、小数表記（例：123.45）、または科学的記数法（例：1.23e3）を使用できます。JSONは Inf、-Inf、NaN をサポートしていません。
  • 論理値（boolean）はRの TRUE と FALSE に似ていますが、小文字の true と false を使用します。

JSONの文字列、数値、論理値は、Rの文字、数値、論理ベクトルに非常によく似ています。主な違いは、JSONのスカラーは単一の値しか表現できないことです。複数の値を表現するには、残りの2つの型のいずれかを使用する必要があります。配列とオブジェクトです。配列とオブジェクトはどちらもRのリストに似ていますが、名前があるかどうかが異なります。

  • 配列（array）は名前のないリストのようなもので、[] で記述されます。例えば [1, 2, 3] は3つの数値を含む配列であり、[null, 1, "string", false] はヌル、数値、文字列、論理値を含む配列です。
  • オブジェクト（object）は名前付きリストのようなもので、{} で記述されます。名前（JSON用語ではキー）は文字列であるため、引用符で囲む必要があります。例えば、{"x": 1, "y": 2} は x を1に、y を2に対応させるオブジェクトです。

JSONには日付や日時を表現するネイティブな方法がないため、それらは文字列として保存されることが多いことに注意してください。それらを正しいデータ構造に変換するには、readr::parse_date() や readr::parse_datetime() を使用する必要があります。同様に、JSONで浮動小数点数を表現する規則は少し曖昧なため、数値が文字列に保存されていることもあります。必要に応じて readr::parse_double() を適用して、正しい変数型を取得してください。

23.5.2 jsonlite

JSONをRのデータ構造に変換するには、Jeroen Ooms氏による jsonlite パッケージをお勧めします。ここでは read_json() と parse_json() という2つの関数のみを使用します。実際には、ディスクからJSONファイルを読み込むために read_json() を使用します。例えば、repurrrsive パッケージは gh_user のソースをJSONファイルとしても提供しており、read_json() で読み込むことができます。

# パッケージ内のjsonファイルへのパス:
gh_users_json()
#> [1] "C:/Users/777mi/AppData/Local/R/win-library/4.5/repurrrsive/extdata/gh_users.json"

# read_json()で読み込む
gh_users2 <- read_json(gh_users_json())

# 以前使用していたデータと同じであることを確認
identical(gh_users, gh_users2)
#> [1] TRUE

この本では、JSONを含む文字列を受け取る parse_json() も使用します。これは簡単な例を生成するのに適しています。まずは、3つの単純なJSONデータセットから始めましょう。数値から始まり、いくつかの数値を配列に入れ、最後にその配列をオブジェクトに入れます。

str(parse_json('1'))
#>  int 1
str(parse_json('[1, 2, 3]'))
#> List of 3
#>  $ : int 1
#>  $ : int 2
#>  $ : int 3
str(parse_json('{"x": [1, 2, 3]}'))
#> List of 1
#>  $ x:List of 3
#>   ..$ : int 1
#>   ..$ : int 2
#>   ..$ : int 3

jsonliteには fromJSON() という別の重要な関数があります。ここではそれを使用していません。なぜなら、これは自動的な簡略化（simplifyVector = TRUE）を行うからです。これは単純なケースではうまく機能することが多いのですが、何が起こっているのかを正確に把握し、より複雑なネスト構造を簡単に扱えるようにするために、自分自身でレクタングリングを行う方が良いと考えています。

23.5.3 レクタングリング・プロセスの開始

ほとんどの場合、JSONファイルは単一の最上位配列を含んでいます。これは、複数の「もの」（例えば複数のページ、複数のレコード、複数の結果）に関するデータを提供するように設計されているためです。この場合、各要素が行になるように tibble(json) からレクタングリングを開始します。

json <- '[
  {"name": "John", "age": 34},
  {"name": "Susan", "age": 27}
]'
df <- tibble(json = parse_json(json))
df
#> # A tibble: 2 × 1
#>   json            
#>   <list>          
#> 1 <named list [2]>
#> 2 <named list [2]>

df |>
  unnest_wider(json)
#> # A tibble: 2 × 2
#>   name    age
#>   <chr> <int>
#> 1 John     34
#> 2 Susan    27

稀なケースですが、JSONファイルが1つの「もの」を表す単一の最上位JSONオブジェクトで構成されている場合があります。この場合は、tibbleに入れる前に、リストでラップしてレクタングリング・プロセスを開始する必要があります。

json <- '{
  "status": "OK",
  "results": [
    {"name": "John", "age": 34},
    {"name": "Susan", "age": 27}
 ]
}
'
df <- tibble(json = list(parse_json(json)))
df
#> # A tibble: 1 × 1
#>   json            
#>   <list>          
#> 1 <named list [2]>

df |>
  unnest_wider(json) |>
  unnest_longer(results) |>
  unnest_wider(results)
#> # A tibble: 2 × 3
#>   status name    age
#>   <chr>  <chr> <int>
#> 1 OK     John     34
#> 2 OK     Susan    27

あるいは、解析されたJSONの内部にアクセスして、実際に必要な部分から始めることもできます。

df <- tibble(results = parse_json(json)$results)
df |>
  unnest_wider(results)
#> # A tibble: 2 × 2
#>   name    age
#>   <chr> <int>
#> 1 John     34
#> 2 Susan    27

23.5.4 練習問題

  1. 以下の df_col と df_row をレクタングリングしてください。これらは、JSONでデータフレームをエンコードする2つの方法を表しています。
    json_col <- parse_json('
      {
        "x": ["a", "x", "z"],
        "y": [10, null, 3]
      }
    ')
    json_row <- parse_json('
      [
        {"x": "a", "y": 10},
        {"x": "x", "y": null},
        {"x": "z", "y": 3}
      ]
    ')

    df_col <- tibble(json = list(json_col))
    df_row <- tibble(json = json_row)

23.6 まとめ

この章では、リストとは何か、JSONファイルからリストを生成する方法、そしてそれらを長方形のデータフレームに変換する方法を学びました。驚くべきことに、必要なのは2つの新しい関数だけです。リスト要素を行に配置する unnest_longer() と、リスト要素を列に配置する unnest_wider() です。リスト列がどれほど深くネストされていても、これら2つの関数を繰り返し呼び出すだけで済みます。

JSONは、ウェブAPIから返される最も一般的なデータ形式です。もしウェブサイトにAPIがないけれど、ウェブサイト上に欲しいデータが見えている場合はどうすればよいでしょうか？ それが次の章のテーマです。HTMLウェブページからデータを抽出する「ウェブスクレイピング」について学びます。
